# Story 5.7 — HITL Request Tool Registration (request_human_input)

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.3 must be `DONE` (MCP Skill Router must exist before new tools are registered).

## Status

`[ ] PENDING`

## Description

Define and register a core system MCP tool named `request_human_input` that any agent persona can call during execution to pause the orchestration loop and ask the human operator a structured question. This is distinct from the manual-mode approval gate in Story 5.5: 5.5 is a fixed checkpoint after every agent output; this tool is an AI-initiated, mid-execution interruption that the agent itself decides to trigger.

## Acceptance Criteria

- [ ] A new skill file `request-human-input.skill.ts` is created and auto-loaded by the Dynamic Registry Loader (5.1).
- [ ] The MCP tool definition exposed to the AI:
  ```json
  {
    "name": "request_human_input",
    "description": "Pause execution and ask the human operator a question. Use when you need a decision, credential, or clarification that you cannot safely infer.",
    "input_schema": {
      "type": "object",
      "properties": {
        "message": {
          "type": "string",
          "description": "The question or prompt to display to the user."
        },
        "options": {
          "type": "array",
          "items": { "type": "string" },
          "description": "Ordered list of labelled options the user can choose from. Provide at least 2. May be empty if free-text is required."
        },
        "context": {
          "type": "string",
          "description": "Optional additional context shown below the question (e.g. a file path, a diff, a command output)."
        },
        "allow_free_text": {
          "type": "boolean",
          "description": "If true, renders a free-text input in addition to the option buttons. Defaults to false."
        }
      },
      "required": ["message", "options"]
    }
  }
  ```
- [ ] The skill's handler does NOT execute immediately. Instead it calls `OrchestrationEventBus.emitHitlRequest(workflowId, payload)` and returns a `Promise` that is stored in a `PendingInputRegistry` (in-memory map: `workflowId → { resolve, reject, payload, expiresAt }`).
- [ ] Registry stores `{ resolve, reject, payload: HITLRequestPayload }` per pending ID. `HITLRequestPayload` shape: `{ toolName: string; proposedOutput: unknown; sessionId: string; workflowExecutionId: string }`.
- [ ] The `PendingInputRegistry` resolves the stored `Promise` when `POST /api/workspaces/:wsId/workflows/:workflowId/respond` is called (implemented in 5.8).
- [ ] The tool is universally available: the `SkillRouter` (5.3) includes it in every agent's tool definition list without requiring explicit opt-in per agent.
- [ ] The tool is excluded from the configurable skill registry UI — it is a system-level tool that cannot be disabled by workspace admins.
- [ ] Zod validates the incoming tool input before the `Promise` is stored.

## Technical Notes

- Sequencing note: `PendingInputRegistry` (this story) stores `{ resolve, reject, payload }` callbacks. Story 5.8 adds the `POST .../respond` endpoint that calls `registry.resolve(payload)`. Story 6.10 adds `GET .../pending-input` that needs the _original request payload_ — update `PendingInputRegistry` to also store `{ resolve, reject, payload }` where `payload` is the HITL request data (tool name, proposed output, context). The shape: `Map<string, { resolve: Function; reject: Function; payload: HITLRequestPayload }>`.

## Security Notes

- The respond endpoint (5.8) must validate the JWT and confirm the responding user is a member of the workspace that owns the workflow. A user from Workspace A must not be able to answer a HITL request belonging to Workspace B.
- `PendingInputRegistry` entries expire after 30 minutes (configurable via `HITL_TIMEOUT_MINUTES`). On expiry, the Promise rejects with `HITL_TIMEOUT` error, which causes the orchestrator (5.8) to transition the workflow to `IDLE` error state.

## Tests to Write

- Tool definition matches Anthropic SDK tool schema format.
- Handler stores resolver in `PendingInputRegistry` and the Promise is pending.
- Resolving via `PendingInputRegistry.resolve(workflowId, answer)` resolves the Promise with the answer.
- Entry expires after timeout → Promise rejects with `HITL_TIMEOUT`.
- Input with empty `options` array and `allow_free_text: false` → Zod validation error.
- Tool appears in every agent's tool list returned by `SkillRouter`.

## Definition of Done

- Tool registered and included in all agent tool definitions. Handler returns a Promise that suspends the agentic loop. Timeout and expiry enforced.
