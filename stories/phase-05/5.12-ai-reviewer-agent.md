## Description

The architecture has only _producer_ agents. A reviewer agent turns the pipeline into a producer/critic loop: after the developer agent's change passes automated verification, a separate reviewer agent — with a distinct persona and a fresh read of the diff — critiques it against the acceptance criteria, the workspace rules, and common defect patterns **before a human sees it**. The human then reviews a pre-reviewed change, which is a large quality multiplier and reduces human review load.

## Acceptance Criteria

### Review State & Flow

- [ ] After `VERIFYING` passes (Story 5.10) and the security gate passes (Story 9.5), the orchestrator transitions `VERIFYING → REVIEWING` when the reviewer is enabled for the workspace.
- [ ] The reviewer agent (`reviewer-agent.md`, `model_tier: standard`) receives a **fresh context**: the git diff, the Jira story acceptance criteria, the workspace rules and constraints/patterns from memory (Story 4.4), and the verification results — but NOT the developer agent's chain-of-thought (independent review, not a rubber stamp).
- [ ] The reviewer must return a structured verdict via a `submit_review` skill (planning-phase-style structured tool, no free text):
  ```typescript
  {
    verdict: 'approve' | 'request_changes';
    findings: Array<{
      severity: 'blocker' | 'major' | 'minor' | 'nit';
      file: string;
      line: number | null;
      issue: string;
      suggestion: string;
      acceptance_criterion?: string;   // which AC this relates to, if any
    }>;
    unmet_acceptance_criteria: string[];  // ACs the reviewer judges not satisfied
  }
  ```

### Routing on Verdict

- [ ] `request_changes` with any `blocker` or `major` finding, OR any `unmet_acceptance_criteria`:
  - Findings are compiled into a revision payload (reuse Story 5.5 builder) and fed back to the **developer** agent: `"An automated reviewer found issues. Address these:\n{findings}"`.
  - Transition `REVIEWING → RUNNING`.
- [ ] `approve` (or only `minor`/`nit` findings remaining):
  - MANUAL mode → `REVIEWING → AWAITING_APPROVAL`. The reviewer's findings (including nits) are attached to the approval payload so the human sees what the reviewer flagged.
  - AUTO mode → `REVIEWING → COMPLETED`.
- [ ] **Review cycle budget**: max `MAX_REVIEW_CYCLES` (default 2, configurable). On exhaustion, stop looping and go to `AWAITING_APPROVAL` (MANUAL) with a `review_unresolved: true` flag, or `COMPLETED` (AUTO) with the outstanding findings recorded — never loop developer↔reviewer indefinitely. Review cycles count against `MAX_LOOP_ITERATIONS` and `MAX_WORKFLOW_TOKENS`.
- [ ] The producer and reviewer must be different agent instances with different personas; the reviewer must never be able to edit files (no `write_file`/`run_terminal_command` in its tool set) — it can only read and submit a review.

### Configuration

- [ ] `GET`/`PUT /api/workspaces/:id/reviewer-settings` (WORKSPACE_OWNER+): enable/disable the reviewer, set `MAX_REVIEW_CYCLES`, and set the blocking severity floor (default: block on `blocker`+`major`).
- [ ] Reviewer disabled → pipeline is `VERIFYING → AWAITING_APPROVAL/COMPLETED` exactly as before this story (no regression).

## Technical Notes

- The reviewer's independent context is the whole point — do not pass the developer's messages. Passing the producer's reasoning biases the critic toward agreement.
- The reviewer uses `standard` tier by default (it needs real judgment), but a workspace optimising cost can drop it to `economy` via agent config.
- All review verdicts are appended to `workflow_executions.history` for later analytics (how often the reviewer catches issues = a quality metric for Story 13.8 eval harness).
- The reviewer's read-only tool restriction is enforced by the `SkillRouter` tool-list assembly for the `REVIEWING` state — the same state-scoped tool-exposure mechanism used by the planning gate (Story 5.6).

## Tests to Write

- Verification passes + reviewer enabled → status transitions to `REVIEWING`.
- Reviewer `request_changes` with a blocker → findings fed to developer, status → `RUNNING`.
- Reviewer flags `unmet_acceptance_criteria` → treated as blocking regardless of finding severities.
- Reviewer `approve` (MANUAL) → `AWAITING_APPROVAL`, findings attached to approval payload.
- Reviewer `approve` (AUTO) → `COMPLETED`.
- Only `nit` findings → not blocking, advances.
- Review cycle budget exhausted → advances with `review_unresolved: true`, no infinite loop.
- Reviewer agent tool set excludes `write_file`/`run_terminal_command`.
- Reviewer context excludes the developer agent's messages.
- Reviewer disabled → pipeline behaves exactly as pre-5.12.

## Definition of Done

- Verified changes are independently critiqued by a read-only reviewer agent before humans see them. Blocking findings and unmet ACs loop back to the developer within budget. Humans receive pre-reviewed changes with the reviewer's notes attached. No infinite producer/critic loops. Tests pass.
