# Story 5.8 — Orchestrator Loop Suspension & Activity Event Stream

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.4 must be `DONE` (base state machine).
- Story 5.7 must be `DONE` (`request_human_input` tool and `PendingInputRegistry`).

## Status

`[ ] PENDING`

## Description

Extend the `WorkflowOrchestrator` agentic loop with two capabilities:

1. **Loop Suspension**: When the AI calls `request_human_input`, the loop awaits the `PendingInputRegistry` Promise rather than continuing. The workflow transitions to `AWAITING_INPUT` state and the context snapshot is persisted so it can survive a process restart.

2. **Activity Event Stream**: During every turn of the agentic loop, emit granular `agent_activity` WebSocket events so the frontend (Story 6.9) can display real-time "what the agent is doing" chips. These events are transient — not persisted to the DB.

## Acceptance Criteria

### Loop Suspension

- [ ] New workflow state added to the `workflow_executions.status` enum: `AWAITING_INPUT`.
- [ ] New state transitions added to `WorkflowOrchestrator`:
  - `RUNNING → AWAITING_INPUT` when `request_human_input` tool call is detected in the LLM's response.
  - `AWAITING_INPUT → RUNNING` when the user's answer arrives and is injected back as the tool result.
  - `AWAITING_INPUT → IDLE` (error) if the HITL request times out (`HITL_TIMEOUT`).
- [ ] When transitioning to `AWAITING_INPUT`:
  1. Serialize the current `messages[]` array (full conversation history) to the `active_sessions.context_snapshot` JSONB column.
  2. Transition DB state to `AWAITING_INPUT`.
  3. Emit WebSocket event: `{ type: 'workflow_state_change', payload: { workflowId, status: 'AWAITING_INPUT' } }`.
  4. The Node.js event loop is NOT blocked — the orchestrator `await`s the `PendingInputRegistry` Promise, which naturally suspends the async function without blocking the process.
- [ ] `POST /api/workspaces/:wsId/workflows/:workflowId/respond`
  - Auth: JWT + workspace membership check.
  - Request body: `{ answer: string }` (Zod validated, non-empty string, max 2000 chars).
  - Calls `PendingInputRegistry.resolve(workflowId, answer)`.
  - Transitions DB state back to `RUNNING`.
  - Returns `{ status: 'resumed' }`.
  - 404 if no pending HITL request exists for this `workflowId`.
- [ ] On resume: the tool result `{ answer }` is appended to `messages[]` as an Anthropic tool_result message, and the agentic loop continues.
- [ ] On process restart: at startup, any `workflow_executions` in `AWAITING_INPUT` state that have no live `PendingInputRegistry` entry are transitioned to `IDLE` with error `PROCESS_RESTARTED` and a WebSocket notification is emitted.

### Activity Event Stream

- [ ] The agentic loop emits `agent_activity` WebSocket events at these points:
  | Event trigger          | `activity_type` | Example `description`              |
  | ---------------------- | --------------- | ---------------------------------- |
  | LLM API call starts    | `llm_thinking`  | `"Agent is reasoning..."`          |
  | LLM API call completes | `llm_response`  | `"Agent produced a response"`      |
  | Tool call starts       | `tool_start`    | `"Reading file src/index.ts"`      |
  | Tool call completes    | `tool_done`     | `"File read successfully"`         |
  | Tool call errors       | `tool_error`    | `"Command failed: exit code 1"`    |
  | HITL request triggered | `hitl_pending`  | `"Agent is asking for your input"` |
- [ ] Event shape:
  ```typescript
  {
    type: 'agent_activity',
    payload: {
      workflowId: string,
      activity_type: 'llm_thinking' | 'llm_response' | 'tool_start' | 'tool_done' | 'tool_error' | 'hitl_pending',
      description: string,          // human-readable, shown in UI chip
      tool_name?: string,           // present for tool_* events
      tool_input_summary?: string,  // short summary, NOT the full input (no secrets)
      timestamp: string             // ISO 8601
    }
  }
  ```
- [ ] `tool_input_summary` must be sanitized: truncate to 120 chars, redact values from keys that match `/password|token|secret|key|credential/i`.
- [ ] Activity events are NOT stored in the DB (transient only). Do not add a table for them.

## Tests to Write

- Agentic loop awaits when `request_human_input` is called → state transitions to `AWAITING_INPUT`.
- `POST .../respond` with valid answer → state returns to `RUNNING`, loop continues with tool result.
- `POST .../respond` with no pending request → 404.
- HITL timeout → state transitions to `IDLE` with `HITL_TIMEOUT` error.
- `agent_activity` event emitted on tool call start and end.
- `tool_input_summary` redacts values matching secret key patterns.
- Process restart with stale `AWAITING_INPUT` record → record transitions to `IDLE`.

## Definition of Done

- Loop suspends correctly without blocking the Node.js process. Context persists through `AWAITING_INPUT`. Activity events emitted at every meaningful loop step. `respond` endpoint resumes the loop cleanly.
