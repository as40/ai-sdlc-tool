# Story 5.14 — Core Workflow Definition Library

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.1 must be `DONE` (registry loader — extended here to also load `workflows/`).
- Story 5.4 must be `DONE` (orchestrator that executes a workflow definition).
- Story 5.13 must be `DONE` (agents the workflows sequence).

## Status

`[ ] PENDING`

## Description

The orchestrator (5.4) is the engine that runs workflows, but no story authors the actual **workflow definitions** — the declarative sequences of stages/agents that make up a repeatable SDLC task. The `workflows/` directory (created empty in 1.1) is not even scanned by the registry loader today. This story extends the registry to load workflow definitions and authors the core set, so a task like "implement this story" runs a known, auditable pipeline rather than being improvised each time.

## Registry Extension

- [ ] Extend `RegistryLoader` (5.1) to also scan `src/workflows/` and populate a new `WorkflowRegistry: Map<string, WorkflowDefinition>` keyed by workflow name.
- [ ] Each workflow is a declarative file (`.md` with YAML frontmatter, or `.yaml`) with: `name`, `description`, `version`, and an ordered `stages[]` list. A stage: `{ id, agent, on_success, on_failure, gate?, max_cycles? }`.
- [ ] `gate` values map to existing states: `plan_approval` (5.6), `verification` (5.10), `security_scan` (9.5), `review` (5.12), `human_approval` (5.5). The orchestrator already implements these gates; the workflow just declares which apply and in what order.
- [ ] `GET /api/registry/workflows` (WORKSPACE_OWNER+) lists loaded workflows. Malformed workflow files are skipped with a warning (do not crash), consistent with 5.1.
- [ ] `POST /api/workspaces/:id/workflows` (Story 5.4) accepts an optional `workflowName`; if omitted, the supervisor selects one. Selected/였-resolved workflow name is recorded on `workflow_executions`.

## Core Workflow Set (5 workflows)

| Workflow file        | Stage sequence (agents + gates)                                                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `implement-story.md` | planner → **plan_approval** → developer → test-generator → **verification** → **security_scan** → reviewer → **review** → **human_approval**     |
| `fix-bug.md`         | debugger (reproduce + diagnose) → developer (fix) → test-generator (regression test) → **verification** → **security_scan** → **human_approval** |
| `write-tests.md`     | test-generator → **verification** → **human_approval**                                                                                           |
| `refactor.md`        | planner → **plan_approval** → developer → **verification** (behavior must stay green) → reviewer → **review** → **human_approval**               |
| `review-only.md`     | reviewer → **review** (read-only; no writes, produces a report, no approval-to-merge)                                                            |

## Acceptance Criteria

- [ ] All 5 workflow files exist in `src/workflows/`, load into `WorkflowRegistry`, and reference only agents that exist in the `AgentRegistry` (5.13) and gates the orchestrator implements.
- [ ] The orchestrator (5.4) executes a loaded `WorkflowDefinition`: it walks `stages[]`, invokes each stage's agent, and applies the declared gate before advancing. `on_failure` routing (e.g. verification fail → back to developer) is honored, bounded by `max_cycles`.
- [ ] In AUTO mode, gates that require a human (`plan_approval`, `human_approval`) are configurable to auto-pass or to still halt; default: `human_approval` always halts in MANUAL, auto-passes in AUTO.
- [ ] A workflow referencing an unknown agent or gate → loader warning, workflow marked invalid and excluded from selection (not a crash).
- [ ] Workflow selection is recorded for audit (ties to Story 10.6 provenance): which workflow ran for a given execution.

## Tests to Write

- All 5 workflow files load into WorkflowRegistry.
- Each workflow references only existing agents and valid gates.
- Malformed workflow / unknown agent → skipped with warning, others still load.
- Orchestrator runs `implement-story` end-to-end (mock agents/gates) hitting stages in order.
- `on_failure` from the verification gate routes back to the developer stage within `max_cycles`.
- `review-only` grants the reviewer no write path.
- Workflow name persisted on the execution record.

## Definition of Done

- The registry loads workflow definitions and the core 5 exist and are valid. The orchestrator executes a declared workflow's stages and gates deterministically. Workflow selection is auditable. Tests pass.
