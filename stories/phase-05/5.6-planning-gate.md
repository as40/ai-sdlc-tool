## Description

In MANUAL execution mode, before the agent touches any file or runs any command, it must first produce a structured execution plan and wait for human approval. This eliminates the most expensive class of wasted loop: a fundamentally wrong approach executed fully before anyone can intervene.

The agent calls a dedicated `submit_execution_plan` tool to output its plan in a structured format. The orchestrator halts at `AWAITING_PLAN_APPROVAL`, presents the plan to the user, and only proceeds to `RUNNING` on explicit approval. If the user rejects the plan, the agent is instructed to revise and resubmit — up to a configurable maximum.

AUTO mode is unaffected: workflows skip `PLANNING` entirely and start at `RUNNING`.

## Acceptance Criteria

### Planning State

- [ ] In MANUAL mode, `WorkflowOrchestrator` transitions: `IDLE → PLANNING` on workflow start.
- [ ] In `PLANNING` state, the orchestrator calls the LLM exactly once with the system prompt + story context but **without** any file-writing skills exposed. Only the `submit_execution_plan` tool is available during this turn.
- [ ] `submit_execution_plan` skill:
  - Input schema (Zod-validated):
    ```typescript
    {
      summary: string;               // 1–3 sentence description of the approach
      steps: Array<{
        order: number;
        description: string;         // what will be done
        files_affected: string[];    // file paths the agent expects to read or write
        risk: 'low' | 'medium' | 'high';
      }>;
      estimated_iterations: number;  // agent's estimate of LLM turns needed
      open_questions: string[];      // anything the agent is uncertain about (may be empty)
    }
    ```
  - Returns `{ received: true }` — acknowledgement only; the orchestrator handles the rest.
  - Validation: `steps` must have 1–20 entries; `summary` max 500 chars; each `description` max 300 chars.
- [ ] If the AI does not call `submit_execution_plan` within 1 LLM turn, the orchestrator transitions the workflow to `IDLE` with error `PLANNING_TIMEOUT` and emits a WebSocket event.
- [ ] After plan is submitted, orchestrator transitions to `AWAITING_PLAN_APPROVAL` and emits:
  ```typescript
  {
    type: 'plan_submitted',
    payload: {
      workflowId: string;
      plan: SubmittedPlan;           // the structured plan object
    }
  }
  ```

### Approval Endpoints

- [ ] `POST /api/workspaces/:wsId/workflows/:workflowId/plan/approve`
  - Auth: JWT + `requireRole('DEVELOPER', 'WORKSPACE_OWNER', 'SUPER_ADMIN')`.
  - Transitions: `AWAITING_PLAN_APPROVAL → RUNNING`.
  - Emits `{ type: 'workflow_state_change', payload: { workflowId, status: 'RUNNING' } }`.
  - All full execution skills (`read_file`, `write_file`, `run_terminal_command`, etc.) are now available to the agent.
  - Returns `{ status: 'approved' }`.
- [ ] `POST /api/workspaces/:wsId/workflows/:workflowId/plan/revise`
  - Auth: JWT + `requireRole('DEVELOPER', 'WORKSPACE_OWNER', 'SUPER_ADMIN')`.
  - Request body: `{ feedback: string }` (Zod, max 1000 chars).
  - Transitions: `AWAITING_PLAN_APPROVAL → PLANNING`.
  - Injects feedback into the agent's next turn as a user message:
    ```
    Your plan was rejected. Feedback: {feedback}
    Please revise your plan and call submit_execution_plan again.
    ```
  - Revision count incremented. Maximum plan revisions: 3 (configurable via `MAX_PLAN_REVISIONS`, default 3). On 4th rejection, workflow transitions to `IDLE` with error `PLAN_REVISION_LIMIT`.
  - Returns `{ status: 'revision_requested', revisionCount: number }`.

### Plan Persistence

- [ ] Migration: add `execution_plan JSONB NULL` column to `workflow_executions` table. Stores the last approved plan.
- [ ] The approved plan object is stored in `workflow_executions.execution_plan` before transitioning to `RUNNING`.
- [ ] `GET /api/workspaces/:wsId/workflows/:workflowId` response includes `execution_plan` field (null if AUTO mode or not yet approved).

## Technical Notes

- The planning turn uses the `OrchestratorSupervisor`'s model tier (economy) — the planner does not need an expensive model.
- `submit_execution_plan` is registered as a skill with `SkillRouter` but is only injected into the tool list during the `PLANNING` state. In `RUNNING` state it must not be available (prevents the agent from re-planning when it should be executing).
- Plan revisions reuse the same `PLANNING` state loop (not a new state). The revision counter is tracked in `workflow_executions.history` JSONB.
- `open_questions` in the plan is surfaced to the user in the `plan_submitted` WS event so they can address them before approving.

## Tests to Write

- MANUAL mode start → status transitions to `PLANNING`.
- AUTO mode start → status transitions directly to `RUNNING` (planning gate skipped).
- Agent calls `submit_execution_plan` → status transitions to `AWAITING_PLAN_APPROVAL`, WS event emitted.
- Agent does not call plan tool in 1 turn → `PLANNING_TIMEOUT`, status → `IDLE`.
- `POST .../plan/approve` → status → `RUNNING`, plan stored in DB, full skill set available.
- `POST .../plan/revise` with feedback → status → `PLANNING`, feedback injected into messages.
- 4th plan revision → `PLAN_REVISION_LIMIT` error, status → `IDLE`.
- VIEWER role on plan/approve → 403.
- `submit_execution_plan` called during `RUNNING` state → `SKILL_NOT_FOUND` error (not in tool list).
- Invalid plan schema (empty steps) → `VALIDATION_ERROR` from SkillRouter.
