# Story 5.16 — Core Tech-Stack Standards (Plugin) Library

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.1 must be `DONE` (PluginRegistry loads `plugins/{stack}/standards.md`).
- Story 5.2 must be `DONE` (prompt compiler merges plugin standards into the system prompt).
- Story 3.3 should be `DONE` (tech-stack scanner produces the stack key used to select a plugin).

## Status

`[ ] PENDING`

## Description

The prompt compiler (5.2) merges `plugins/{tech-stack}/standards.md` into every system prompt, and the PluginRegistry (5.1) loads them — but **no story authors the standards content**. Without it, the compiler's plugin section is always empty and the AI writes stack-generic code. This story authors the core per-stack standards library: the framework-specific idioms, conventions, and pitfalls that make AI output idiomatic and correct for the detected stack. This is what makes the difference between "code that compiles" and "code that looks like it belongs in this codebase."

## Standards Set (7 plugins)

| Plugin dir            | Covers                                                                                                 |
| --------------------- | ------------------------------------------------------------------------------------------------------ |
| `plugins/generic/`    | Fallback used when the scanner can't map a stack. Language-agnostic quality/security/testing guidance. |
| `plugins/node/`       | Node.js runtime idioms: async/await, error handling, streams, no blocking calls.                       |
| `plugins/typescript/` | Strict TS: no `any`, `unknown` + narrowing, discriminated unions, type-only imports.                   |
| `plugins/react/`      | Hooks rules, component structure, state management, controlled inputs, accessibility basics.           |
| `plugins/python/`     | PEP 8, type hints, virtualenv, exceptions, no mutable default args.                                    |
| `plugins/java/`       | Package structure, interfaces over impls, streams, null-safety (Optional).                             |
| `plugins/go/`         | Idiomatic error handling, no naked returns, goroutine/context patterns.                                |

## Acceptance Criteria

- [ ] Each plugin directory contains a `standards.md` loaded by the PluginRegistry (5.1), keyed by the stack name the scanner (3.3) emits.
- [ ] Each `standards.md` is **focused and token-lean** — it is merged into every system prompt for that stack, so it must be high-signal (idioms, do/don't, common pitfalls), not an exhaustive style guide. Target a hard ceiling well under the compiler's `MAX_SYSTEM_PROMPT_TOKENS` budget (5.2) so the plugin section never dominates the prompt.
- [ ] Standards reference, but do not duplicate, the repo-level `.rules/*` and the immutable system header — DRY across the prompt layers (header = universal law, standards = stack idioms, rules = repo policy).
- [ ] Multi-stack workspaces: when the scanner detects more than one stack (e.g. a TS React frontend + a Python backend), the compiler (5.2) can merge multiple plugin standards; this story ensures each stands alone and composes without contradiction.
- [ ] Unknown/unsupported stack → compiler falls back to `plugins/generic/standards.md` (already the graceful-skip path in 5.2, now with real fallback content).
- [ ] A `plugins/README.md` documents the authoring contract so new stack plugins can be added later (extension point, aligns with Story 13.x customization).

## Tests to Write

- Each plugin `standards.md` loads into the PluginRegistry keyed by stack name.
- Each standards file is under the per-plugin token ceiling.
- Compiler (5.2) with `stack=react` → compiled prompt contains the react standards section.
- Unknown stack → compiler uses `generic` standards, no error.
- Multi-stack (typescript + python) → both sections merged, prompt stays within `MAX_SYSTEM_PROMPT_TOKENS` (truncation order from 5.2 respected).
- No standards file duplicates the immutable system header content (DRY check).

## Definition of Done

- The core per-stack standards library exists and loads, giving the prompt compiler real content to merge. Output is idiomatic to the detected stack. Files are token-lean, DRY across prompt layers, and compose for multi-stack workspaces with a generic fallback. Tests pass.
