# Story 5.4 ‚Äî State-Machine Workflow Orchestrator

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.2, 5.3, 2.5 must be `DONE`.

## Status

`[ ] PENDING`

## Description

Build the state-driven pipeline engine that manages the lifecycle of an AI workflow execution from IDLE through to COMPLETED, routing tasks to the correct agent at each step.

## Acceptance Criteria

- [ ] `WorkflowOrchestrator` manages `workflow_executions` state transitions:
  - `IDLE ‚Üí RUNNING` (on start ‚Äî AUTO mode)
  - `IDLE ‚Üí PLANNING` (on start ‚Äî MANUAL mode; Story 5.6 extends this state)
  - `RUNNING ‚Üí AWAITING_APPROVAL` (after agent produces output requiring human review)
  - `AWAITING_APPROVAL ‚Üí RUNNING` (on approval)
  - `AWAITING_APPROVAL ‚Üí REVISION_REQUESTED` (on reject with feedback)
  - `REVISION_REQUESTED ‚Üí RUNNING` (feeds revision payload back to agent)
  - `RUNNING ‚Üí COMPLETED` (final agent done)
  - `RUNNING ‚Üí IDLE` (on error/cancel)
- [ ] The following enum values are added to the `workflow_executions.status` enum **now** (in this story's migration) so later stories do not each ship a separate enum migration. Each is implemented by the story noted; this story only defines the enum values and leaves the transitions unimplemented:
  - `'PLANNING'`, `'AWAITING_PLAN_APPROVAL'` ‚Äî implemented by Story 5.6 (planning gate).
  - `'VERIFYING'` ‚Äî implemented by Story 5.10 (self-verification quality gate). Flow: `RUNNING ‚Üí VERIFYING` when the agent signals completion, then `VERIFYING ‚Üí AWAITING_APPROVAL`/`COMPLETED` on pass or `VERIFYING ‚Üí RUNNING` on failure.
  - `'REVIEWING'` ‚Äî implemented by Story 5.12 (AI reviewer agent). Flow: `VERIFYING ‚Üí REVIEWING ‚Üí AWAITING_APPROVAL`.
  - `'AWAITING_INPUT'` ‚Äî implemented by Story 5.8 (loop suspension).
- [ ] `POST /api/workspaces/:id/workflows` ‚Äî start a new workflow.
  - Request: `{ jiraStoryId?: string; userPrompt: string; executionMode: 'AUTO' | 'MANUAL'; agentName?: string }`.
  - Creates `workflow_executions` record.
  - Routes the prompt to the `OrchestratorSupervisor` agent.
  - In MANUAL mode: initial status is `PLANNING` (Story 5.6 handles the planning gate ‚Äî set status but do not block execution here).
- [ ] `OrchestratorSupervisor` agent: decides which specialist agent to invoke based on the task. Defined with `model_tier: economy` in its `.md` file ‚Äî routing decisions use the workspace's economy-tier model, whatever provider that is.
- [ ] `OrchestratorSupervisor` must return a structured JSON response: `{ agentName: string; confidence: number }` where `confidence` is 0‚Äì1. Use tool call `submit_routing_decision` (registered as a planning-phase-only skill, similar to `submit_execution_plan` in Story 5.6) so the response is always structured and never free text.
- [ ] **Confidence-based escalation**: if `confidence < ROUTING_CONFIDENCE_THRESHOLD` (default `0.6`, configurable via env var), the orchestrator does NOT proceed to route the task. Instead:
  1. Emits WS event: `{ type: 'routing_uncertain', payload: { workflowId, suggestedAgent: agentName, confidence, reason } }`.
  2. Transitions to `AWAITING_INPUT` (Story 5.8 mechanism) with a HITL message: `"I'm not confident which agent should handle this task (confidence: {N}%). I'm suggesting '{agentName}'. Confirm, or type the name of the agent you want instead."`.
  3. On human response: use the confirmed or overridden agent name and proceed to `RUNNING`.
  4. On timeout: fall back to the suggested agent and log a warning.
- [ ] Migration: add `tier VARCHAR(20) NOT NULL DEFAULT 'default' CHECK (tier IN ('default','economy','standard','premium'))` column to the `ai_configurations` table (Story 2.5 owns the table; this story extends it).
- [ ] Migration: add `total_tokens_used INTEGER NOT NULL DEFAULT 0` column to `workflow_executions` table.
- [ ] **Token budget per workflow**: after every LLM response, add `usage.input_tokens + usage.output_tokens` (normalised from the provider's usage object) to `workflow_executions.total_tokens_used`. Before each LLM call, check: if `total_tokens_used >= MAX_WORKFLOW_TOKENS` (default `200000`, configurable via env var), abort the loop ‚Äî transition workflow to `IDLE` with error `TOKEN_BUDGET_EXCEEDED` and emit WS event `{ type: 'token_budget_exceeded', payload: { workflowId, tokensUsed } }`. This cap prevents runaway cost on stuck workflows even if the anti-loop middleware (5.9) has not yet fired.
- [ ] `LLMGateway` resolves the target AI config by querying: `SELECT * FROM ai_configurations WHERE workspace_id = ? AND tier = ?` using the agent's `model_tier`. Falls back to `tier = 'default'` if no config is found for the requested tier. If no `default` config exists either, the gateway returns an error `NO_AI_CONFIG`.
  - This design is provider-agnostic. The workspace owner maps tiers to their actual models (Claude, OpenAI, Gemini, local Ollama, etc.) through the AI settings UI. The gateway never hardcodes a model name or provider.
- [ ] `LLMGateway` routes the call to the correct provider adapter based on `ai_configurations.provider`:
  - `'anthropic'` ‚Üí `@anthropic-ai/sdk`
  - `'openai'` ‚Üí `openai` npm package
  - `'custom'` / `isLocal: true` ‚Üí OpenAI-compatible HTTP client pointed at `baseUrl`
  - Unknown provider ‚Üí error `UNSUPPORTED_PROVIDER`
- [ ] Provider adapters share a common interface: `call(messages, tools, systemPrompt): Promise<LLMResponse>`. All orchestration logic above the gateway is provider-neutral.
- [ ] The gateway exposes a single pre-send hook point (`beforeProviderCall(payload)`) so Story 5.11 (DLP redaction) can scrub secrets/PII from the outbound payload before it reaches a cloud provider. In this story the hook is a pass-through no-op; do not implement redaction here, but the seam must exist.
- [ ] **Prompt caching** (Anthropic provider only): when the resolved provider is `anthropic`, mark the compiled system prompt block and the RAG context block with `cache_control: { type: "ephemeral" }`. Other providers do not receive this field. Cache anchors are placed only on these two largest, stable blocks.
- [ ] **Rate-limit fallback**: if the provider returns HTTP 429 (rate limited) or 529 (overloaded), `LLMGateway` automatically retries the call **once** using the next lower tier config:
  - `premium` ‚Üí retry with `standard` tier config
  - `standard` ‚Üí retry with `economy` tier config
  - `economy` ‚Üí no lower tier; fail with original error `RATE_LIMITED`
  - If the fallback tier config also does not exist in the workspace, fail immediately with `RATE_LIMITED` (do not chain further).
  - Log a WARNING on fallback: `"Provider rate-limited on {tier} tier, retrying with {fallback_tier} tier for this call"`.
  - Fallback is **per-call only** ‚Äî the next call in the same loop still tries the originally requested tier first. The fallback is not permanent and must not be stored to DB.
- [ ] Workflow step execution:
  1. Compile system prompt (story 5.2).
  2. Build context (RAG search from embeddings, story 4.1).
  3. Call LLM with system prompt + context + tools.
  4. Execute any tool calls returned by the AI (story 5.3 SkillRouter).
  5. Loop until AI signals completion or AWAITING_APPROVAL.
- [ ] All state transitions persisted to DB.
- [ ] State change events broadcast via WebSocket: `{ type: 'workflow_state_change', payload: { workflowId, status } }`.
- [ ] `GET /api/workspaces/:id/workflows/:id` ‚Äî get workflow status and history.
- [ ] `DELETE /api/workspaces/:id/workflows/:id` ‚Äî cancel a running workflow (transitions to IDLE).

## Technical Notes

- The LLM Gateway service (stub from 2.5) is fully implemented here. It is **provider-agnostic** ‚Äî it does not call Anthropic directly; it routes through the appropriate provider adapter based on the workspace's `ai_configurations` record.
- Agentic loop termination: for Anthropic, `stop_reason === 'end_turn'`; for OpenAI-compatible providers, `finish_reason === 'stop'`. The provider adapter normalises this to a shared `LLMResponse` type before returning.
- Max iterations: 20 (configurable via `MAX_LOOP_ITERATIONS` env var). Exceeding returns an error state.
- **Tier resolution rationale**: `OrchestratorSupervisor` uses `economy` tier because routing/classification does not need the most capable model. Specialist agents (code writing, test generation) use `standard`. `premium` is reserved for agents that genuinely need deeper reasoning. The workspace owner decides what model sits at each tier ‚Äî the system doesn't prescribe it.
- **Prompt caching** is an Anthropic-specific optimization. The gateway applies it only when the provider is `anthropic` ‚Äî it must never send `cache_control` to OpenAI or other providers as it will cause errors.
- No env vars for model names. Model names come from the DB (`ai_configurations.modelName`) ‚Äî they are user-configured, not hardcoded.
- **Confidence-based escalation rationale**: the `OrchestratorSupervisor` is a cheap model making a routing call. Routing mistakes are expensive (wrong agent runs for 20 turns). Asking the human for a 5-second confirmation when confidence is low costs far less than recovering from a misrouted workflow.
- **Token budget vs. max iterations**: these two caps work together, not in place of each other. Max iterations (20) stops looping agents. Token budget (`MAX_WORKFLOW_TOKENS`) stops expensive short loops that burn tokens on large context inputs per turn. Set both.
- **Rate-limit fallback rationale**: per-call, not permanent, so a brief capacity issue on a premium model doesn't permanently degrade the session. The fallback logs a WARNING so operators can detect systemic rate-limit patterns in their monitoring.
- `ROUTING_CONFIDENCE_THRESHOLD` and `MAX_WORKFLOW_TOKENS` must be documented in `.env.example`.
- Schema dependency: `workflow_executions` table (created in Story 1.2) is missing two columns that are added by later stories: `pr_url TEXT` (added by Story 8.3 migration) and `history JSONB` (added by Story 5.5 migration). Story 5.4 may reference these columns in comments but must not rely on them being present until those migrations run.
- `active_sessions` table referenced in Story 5.8 is defined in Story 12.1. Story 5.4 should not reference `active_sessions` directly.

## Tests to Write

- Start workflow AUTO mode ‚Üí status transitions to RUNNING.
- Start workflow MANUAL mode ‚Üí status transitions to PLANNING (Story 5.6 extends from here).
- AUTO mode: workflow reaches COMPLETED without human intervention (mock LLM).
- MANUAL mode: workflow halts at AWAITING_APPROVAL.
- Cancel workflow ‚Üí status transitions to IDLE.
- Max iterations exceeded ‚Üí workflow errors out.
- Workspace has `economy` tier config ‚Üí gateway uses that config's provider and modelName.
- Workspace has no `economy` config ‚Üí gateway falls back to `default` config.
- Workspace has no AI config at all ‚Üí gateway returns `NO_AI_CONFIG` error.
- Provider is `anthropic` ‚Üí `cache_control` applied to system + RAG blocks.
- Provider is `openai` ‚Üí `cache_control` NOT sent (would cause API error).
- Provider is `custom` with `isLocal: true` and `baseUrl` set ‚Üí request routed to that URL via OpenAI-compatible adapter.
- `OrchestratorSupervisor` returns `confidence >= 0.6` ‚Üí workflow proceeds to `RUNNING` without HITL.
- `OrchestratorSupervisor` returns `confidence < 0.6` ‚Üí workflow transitions to `AWAITING_INPUT`, WS `routing_uncertain` event emitted.
- Human confirms suggested agent ‚Üí workflow resumes with that agent.
- Human overrides with different agent name ‚Üí workflow routes to the overridden agent.
- Routing uncertain timeout ‚Üí workflow falls back to suggested agent, warning logged.
- Total tokens reach `MAX_WORKFLOW_TOKENS` before loop completes ‚Üí `TOKEN_BUDGET_EXCEEDED`, workflow ‚Üí `IDLE`.
- Total tokens below limit ‚Üí budget check passes, loop continues.
- `total_tokens_used` incremented in DB after each LLM response.
- Provider returns 429 on `standard` tier ‚Üí gateway retries with `economy` tier config (if it exists).
- Provider returns 429 on `economy` tier (lowest) ‚Üí fails immediately with `RATE_LIMITED`.
- Rate-limit fallback is per-call only ‚Äî next call in same loop retries `standard` first.
- Provider returns 429 but no lower tier config exists ‚Üí fails with `RATE_LIMITED` without further retry.

## Definition of Done

- State machine transitions work correctly. WebSocket events emitted on each state change. LLM gateway calls the configured model.
