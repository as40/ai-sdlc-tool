# Story 5.9 — Anti-Loop Interception Middleware

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.7 must be `DONE` (`request_human_input` tool must be registered to inject).
- Story 5.8 must be `DONE` (loop suspension must work before forced injection has any effect).

## Status

`[ ] PENDING`

## Description

Implement a recursion-detection middleware that sits inside the `WorkflowOrchestrator` agentic loop. It detects when the AI is stuck repeating the same failing tool call and forcibly injects a `request_human_input` call to break the loop. This prevents infinite retry spirals (e.g., a test that always fails, a file that cannot be written) from consuming tokens and running up cost.

## Acceptance Criteria

- [ ] `AntiLoopMiddleware` is a stateful guard instantiated once per workflow execution (not a singleton).
- [ ] After every tool call within the agentic loop, `AntiLoopMiddleware.record(toolName, toolInput, toolResult)` is called.
- [ ] Repetition detection logic:
  - Compute a **call fingerprint**: `SHA-256(toolName + ":" + JSON.stringify(sortedKeys(toolInput)))`. Sorting keys ensures `{ a:1, b:2 }` and `{ b:2, a:1 }` produce the same fingerprint.
  - Maintain a rolling window of the last 10 fingerprints in-memory per execution.
  - If the **same fingerprint appears 3 times in the last 10 calls**, the pattern is considered stuck.
  - The threshold (3 repetitions, 10-call window) is configurable via env vars `ANTI_LOOP_THRESHOLD` and `ANTI_LOOP_WINDOW`.
- [ ] When the stuck pattern is detected:
  1. Emit an `agent_activity` event: `{ activity_type: 'tool_error', description: 'Detected repetitive failure — escalating to human' }`.
  2. Override the next AI turn: inject a `system` message into `messages[]`:
     ```
     SYSTEM OVERRIDE: You have called tool "{toolName}" with the same arguments {ANTI_LOOP_THRESHOLD} times and it has failed each time. You MUST call request_human_input now to ask the operator for guidance. Do not attempt the same command again.
     ```
  3. After the override message is injected, the middleware sets a `forced` flag so it does not inject again for the same fingerprint within the same execution.
- [ ] If the AI ignores the override and repeats the same tool call a 4th time (i.e. `forced` was set and the fingerprint appears again), the middleware forcibly inserts a synthetic `request_human_input` tool call into the `messages[]` array with:
  - `message`: `"The agent has been stuck repeating the same failing command. How would you like to proceed?"`
  - `options`: `["Skip this step and continue", "Abort the workflow", "Retry with a different approach"]`
  - `allow_free_text`: `true`
    This synthetic call triggers the loop suspension (5.8) as if the AI itself had called the tool.
- [ ] The middleware does NOT track repetitions across different workflow executions. State is entirely in-process and discarded when the execution ends.
- [ ] Both the threshold and window are validated at startup: `ANTI_LOOP_THRESHOLD` must be 2–10, `ANTI_LOOP_WINDOW` must be 5–50.

## Tests to Write

- Same tool + same args called 3× → override system message injected.
- Same tool + different args 3× → NO injection (different fingerprints).
- Override injected → AI ignores it and repeats → synthetic `request_human_input` inserted.
- `forced` flag prevents double injection on 3rd repeat.
- Fingerprint computation is order-independent (`{ a, b }` === `{ b, a }`).
- Threshold and window configurable via env vars.
- Invalid env var values → startup validation error logged, safe defaults applied.

## Definition of Done

- Anti-loop guard detects repetitive tool calls. Override message injected at threshold. Synthetic HITL call inserted if override is ignored. No false positives when same tool used with genuinely different arguments.
