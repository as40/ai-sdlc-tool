# Story 5.10 — Self-Verification Quality Gate

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.4 must be `DONE` (`VERIFYING` enum value defined, workflow loop to instrument).
- Story 5.3 must be `DONE` (`run_terminal_command` skill used to execute checks).
- Story 3.3 must be `DONE` (tech-stack scanner tells us which test/lint/typecheck commands apply).
- Story 9.1 should be `DONE` (checks run inside the sandbox). If no sandbox is available, verification runs in a restricted host subprocess with the same command allow-list.

## Status

`[ ] PENDING`

## Description

The AI must prove its own work before a human ever sees it. Today the agent writes code and jumps straight to `AWAITING_APPROVAL`. This story inserts a mandatory `VERIFYING` state between "agent thinks it is done" and "human reviews": the orchestrator runs the workspace's test, lint, and type-check commands, and the workflow cannot advance until they pass (or the agent has exhausted its self-correction budget). This is the single largest lever on perceived output quality — the human reviews verified, green code instead of raw AI output.

## Acceptance Criteria

### Verification State

- [ ] When the agent signals completion (produces final output with no further tool calls, or calls a `signal_ready_for_review` skill), the orchestrator transitions `RUNNING → VERIFYING` instead of going directly to `AWAITING_APPROVAL`/`COMPLETED`.
- [ ] In `VERIFYING`, the orchestrator runs the configured verification commands **in order**, stopping at the first failure:
  1. Type-check (e.g. `tsc --noEmit`)
  2. Lint (e.g. `eslint .`)
  3. Tests (e.g. `pnpm test`)
- [ ] Verification commands are resolved from a `verification_profile` derived from the tech stack (Story 3.3) with sensible per-stack defaults, overridable in workspace settings. Commands run through the same sandboxed `run_terminal_command` path (allow-listed, no shell injection).
- [ ] Each command's pass/fail, exit code, and truncated output are recorded to `workflow_executions.history`.
- [ ] Emit `agent_activity` events for each check: `{ activity_type: 'verify_start' | 'verify_pass' | 'verify_fail', description }`.

### Pass / Fail Routing

- [ ] All checks pass:
  - AUTO mode → `VERIFYING → COMPLETED` (or `VERIFYING → REVIEWING` if Story 5.12 reviewer is enabled).
  - MANUAL mode → `VERIFYING → AWAITING_APPROVAL` (or `→ REVIEWING` first if 5.12 enabled).
- [ ] A check fails:
  - The failing command's output (truncated via the Story 5.3 truncation util) is fed back to the agent as a user message: `"Verification failed at step {step}. Fix the following and continue:\n{output}"`.
  - Transition `VERIFYING → RUNNING`. The agent gets another turn to fix its own work.
- [ ] **Self-correction budget**: max `MAX_VERIFY_RETRIES` cycles (default 3, configurable). On exhaustion, transition to `AWAITING_APPROVAL` in MANUAL mode with a clear banner (`verification_unresolved: true`) so the human knows checks are red, or to `IDLE` with error `VERIFICATION_FAILED` in AUTO mode. AUTO mode must never emit unverified code as `COMPLETED`.
- [ ] The self-correction cycles count against the workflow's overall `MAX_LOOP_ITERATIONS` and `MAX_WORKFLOW_TOKENS` budgets (Story 5.4) — verification cannot be used to bypass those caps.

### Configuration

- [ ] `GET /api/workspaces/:id/verification-profile` and `PUT /api/workspaces/:id/verification-profile` (WORKSPACE_OWNER+): read/update the ordered command list and enable/disable each check.
- [ ] A workspace owner can disable a check (e.g. no type-check for a JS-only repo). At least one check must remain enabled; disabling all returns 400.

## Technical Notes

- Verification runs in the sandbox (Story 9) when available so the host is never exposed to arbitrary test code. The command allow-list is per-workspace and never accepts free-form commands from the agent — the agent cannot alter which verification commands run.
- Keep the fed-back failure output tight (truncate to `MAX_TOOL_OUTPUT_TOKENS`) — a 5000-line test log will blow the context and the token budget.
- This gate composes with Story 9.5 (security scan) and Story 5.12 (reviewer agent): the full pre-human pipeline is `VERIFYING (tests/lint/types) → security scan gate (9.5) → REVIEWING (5.12) → AWAITING_APPROVAL`.

## Tests to Write

- Agent signals done → status transitions to `VERIFYING`, not directly to `AWAITING_APPROVAL`.
- All checks pass (AUTO) → `COMPLETED`.
- All checks pass (MANUAL) → `AWAITING_APPROVAL`.
- Type-check fails → failure fed back, status → `RUNNING`, agent gets another turn.
- Check fails 3× (retry budget exhausted, MANUAL) → `AWAITING_APPROVAL` with `verification_unresolved: true`.
- Check fails, retry budget exhausted (AUTO) → `IDLE` with `VERIFICATION_FAILED` (never `COMPLETED`).
- Verification retries count against `MAX_LOOP_ITERATIONS`.
- Failure output truncated before feed-back.
- Disabling all checks → 400.
- Owner overrides default commands → new commands used.

## Definition of Done

- No workflow reaches a human (MANUAL) or `COMPLETED` (AUTO) with failing verification unless the retry budget is exhausted and explicitly flagged. Checks run sandboxed. Failures feed back for self-correction within budget. Tests pass.
