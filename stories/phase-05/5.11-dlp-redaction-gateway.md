# Story 5.11 — Sensitive-Data Redaction (DLP) Before LLM Calls

## Phase

Phase 5: Core AI Orchestration Engine

## Dependencies

- Story 5.4 must be `DONE` (`LLMGateway` with the `beforeProviderCall` hook seam).
- Story 2.5 must be `DONE` (per-config `provider` / `isLocal` — redaction policy differs for cloud vs local).

## Status

`[ ] PENDING`

## Description

Every time the orchestrator builds context (code chunks, file contents, env values, Jira data) and sends it to a **cloud** model provider, it may be shipping secrets, credentials, PII, or proprietary data to a third party. This is the risk that silently kills enterprise deals: a security team discovers customer data or API keys were sent to an external LLM. This story adds a Data Loss Prevention layer in the `LLMGateway`'s pre-send hook that detects and redacts sensitive content from the outbound payload before it leaves the platform, with a full audit of what was redacted.

## Acceptance Criteria

### Redaction Engine

- [ ] `DLPRedactor.redact(payload, policy): { redacted: LLMPayload; report: RedactionReport }` scans all outbound message content (system prompt, context blocks, user messages, tool results) before the provider call.
- [ ] Detection covers, at minimum:
  - Secrets: API keys, bearer tokens, private keys (`-----BEGIN ... PRIVATE KEY-----`), connection strings, AWS/GCP/Azure credential formats.
  - PII: emails, phone numbers, national IDs (SSN-format), credit-card-format numbers (Luhn-validated to cut false positives).
  - Custom patterns: workspace-defined regex list (e.g. internal customer-ID format).
- [ ] Each detected span is replaced with a stable placeholder token: `[REDACTED:SECRET:{n}]`, `[REDACTED:PII_EMAIL:{n}]`, etc. Placeholders are consistent within a single call so the model can still reason about "the same value" without seeing it.
- [ ] `RedactionReport`: `{ count: number; byType: Record<string, number>; spans: Array<{ type, placeholder }> }` — records what was redacted **but never the original values**.

### Policy by Provider

- [ ] Redaction policy is resolved per LLM call from the resolved `ai_configurations` record:
  - Cloud provider (`isLocal: false`): redaction **enforced** — cannot be turned off below the org policy floor.
  - Local/self-hosted (`isLocal: true`): redaction **off by default** (data never leaves the perimeter), but workspace owner may opt in.
- [ ] `enforcement_mode` per workspace (WORKSPACE_OWNER+, but bounded by an org-level floor set in Story 10.5/11):
  - `block`: if any secret is detected, **abort the call** with error `DLP_BLOCKED` rather than redact-and-send (for the most sensitive workspaces).
  - `redact`: replace and proceed (default for cloud).
  - `off`: only permitted when `isLocal: true`.
- [ ] Secret detection in `block` mode never falls through to sending — fail closed.

### Audit

- [ ] Every call that triggered redaction writes a `dlp_events` row: `id, workspace_id, workflow_execution_id, provider, redaction_count, by_type JSONB, enforcement_mode, created_at`. Original sensitive values are never stored.
- [ ] `GET /api/workspaces/:id/dlp-events` (WORKSPACE_OWNER+) — audit trail of what categories of sensitive data were being sent to which providers, for compliance review.
- [ ] If DLP is in `redact` mode and redaction count is high (configurable threshold), emit a `dlp_warning` WS event so operators notice a workflow leaking large amounts of sensitive context.

## Technical Notes

- Runs inside `LLMGateway.beforeProviderCall` (seam from Story 5.4). It is the last step before the provider adapter serialises the request — nothing bypasses it.
- Prefer high-precision detectors (Luhn for cards, entropy + known-prefix heuristics for keys) to keep false positives low; over-redaction degrades model output quality.
- Redaction must be idempotent and must not corrupt code structure — replace only the sensitive span, keep surrounding syntax intact.
- The reverse mapping (placeholder → original) is held **in-memory for the duration of the single call only**, solely so tool results referencing a redacted value stay consistent. It is discarded when the call returns and is never persisted.
- This composes with Story 5.8's activity-stream redaction (which protects UI display); 5.11 protects the outbound provider payload. Different surfaces, both required.

## Tests to Write

- Payload with an API key → key replaced with `[REDACTED:SECRET:0]`, original absent from outbound payload.
- Same value appears twice → same placeholder both times (consistency).
- Credit-card-format number failing Luhn → NOT redacted (false-positive guard).
- Valid card number → redacted as PII.
- Cloud provider + `off` mode requested → rejected (cannot disable below floor).
- Local provider + `off` mode → payload sent unredacted.
- `block` mode + secret detected → call aborts with `DLP_BLOCKED`, provider never called.
- `dlp_events` row written with counts but assert original values absent from row.
- Custom workspace regex → matched and redacted.
- Redaction preserves surrounding code syntax (structure intact).

## Definition of Done

- No secret or PII leaves the platform to a cloud provider without redaction (or a hard block). Local models can opt out. Every redaction event is audited without storing sensitive values. Detectors tuned to avoid over-redaction. Tests pass.
