## Status

`[ ] PENDING`

## Description

Regulated enterprises (finance, healthcare, government) must answer: "for this line of code that reached production, which AI model produced it, from what prompt, who reviewed it, and who approved it?" This story creates an immutable, exportable provenance record tying every AI-authored change to its full lineage, plus a tamper-evident audit log suitable for SOC 2 / ISO 27001 evidence.

## Acceptance Criteria

### Provenance Record

- [ ] `change_provenance` table, one row per AI-authored change set (per workflow completion):
      `id, workspace_id, workflow_execution_id, jira_story_id, model_name, provider, prompt_hash, system_prompt_hash, files_changed JSONB, commit_sha TEXT NULL, pr_url TEXT NULL, verification_result JSONB, security_scan_report_id UUID NULL, reviewer_verdict JSONB NULL, approvals JSONB, initiated_by UUID, completed_at TIMESTAMPTZ, prev_hash TEXT, record_hash TEXT`.
- [ ] `record_hash = SHA-256(canonical(row_without_hash) + prev_hash)` — each record chains to the previous record for its workspace, making the log **tamper-evident** (altering a past record breaks the chain).
- [ ] The record links out to the security scan (Story 9.5), reviewer verdict (Story 5.12), verification result (Story 5.10), and approval trail (Story 13.7) so a single provenance row resolves the full quality/governance lineage of a change.
- [ ] Prompt content is stored by **hash** by default (privacy/size); full prompt text retention is an opt-in per-workspace setting for orgs whose policy requires it (bounded by DLP — Story 5.11 redaction applies before storage).

### Audit Log

- [ ] `audit_events` append-only table capturing security-relevant actions: config changes, AI-config/key changes, role changes, approvals/rejections, budget changes, hard-purge (Story 12.2), policy changes. Fields: `id, workspace_id, actor_id, action, target_type, target_id, metadata JSONB, ip, created_at, prev_hash, record_hash` (same hash-chaining).
- [ ] Writes are append-only at the application layer; no update/delete endpoints exist. A background integrity checker can verify the hash chain and flag breaks.

### Export & Access

- [ ] `GET /api/workspaces/:id/provenance` and `.../audit-events` (WORKSPACE_OWNER+ / dedicated `COMPLIANCE_AUDITOR` role) — filterable by date range, story, actor, action.
- [ ] `POST /api/workspaces/:id/compliance-export` — generates a signed export bundle (JSON + CSV) of provenance + audit events for a period, with the chain-verification result included, for handing to auditors.
- [ ] A read-only `COMPLIANCE_AUDITOR` role (RBAC extension) can read audit/provenance across the workspace but cannot modify anything or view secrets.

## Technical Notes

- Hash-chaining gives tamper-evidence without external infrastructure; if an org needs stronger guarantees, the `record_hash` sequence can be periodically anchored (e.g. checkpoint hash written to an external WORM store) — leave a hook, don't build the anchor here.
- Capture provenance at workflow COMPLETED from data already gathered by the pipeline (verification, scan, review, approvals) — do not re-run anything.
- The audit log must never store secret values (keys are already encrypted per Story 1.3; audit records reference the fact of a change, not the value).
- Provenance/audit rows are exempt from the Story 12.2 hard-purge of session data — compliance records outlive sessions. Make this explicit so a purge doesn't erase the audit trail.

## Tests to Write

- Workflow completes → provenance row created with model, prompt hash, files, approvals, links to scan/review/verification.
- Provenance hash-chain valid across multiple records; altering a past row → chain verification fails.
- Audit events append-only; no update/delete path exists.
- `COMPLIANCE_AUDITOR` role → can read audit/provenance, cannot modify, cannot read secrets.
- Compliance export bundle includes chain-verification result and is well-formed.
- Prompt stored as hash by default; opt-in full retention passes through DLP redaction first.
- Hard-purge (12.2) does not delete provenance/audit rows.
- No secret value ever appears in an audit or provenance record.

## Definition of Done

- Every AI-authored change has an immutable, tamper-evident provenance record resolving model, prompt, verification, scan, review, and approvals. Security-relevant actions are captured in an append-only hash-chained audit log. Auditors can export signed evidence bundles. Compliance records survive session purges. Tests pass.
