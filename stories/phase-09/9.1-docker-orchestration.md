# Story 9.1 — Multi-Repo Docker Orchestration

## Phase

Phase 9: Containerized Sandbox Runtime Environment

## Dependencies

- Story 3.3 (tech stack scanner), 3.4 (env gateway) must be `DONE`.

## Status

`[ ] PENDING`

## Description

Trigger the AI DevOps Agent to analyze tech-stack metadata and generate a unified `docker-compose.yml` for all workspace repos, networking them together.

## Acceptance Criteria

- [ ] `POST /api/workspaces/:id/sandbox/generate-compose` — triggers compose generation.
- [ ] `DevOpsAgent` (new agent definition) reads `repositories.tech_stack` for all workspace repos.
- [ ] Generates a `docker-compose.yml` at the root of the Parent Workspace Directory.
- [ ] Compose file includes:
  - One service per repo with appropriate base image (e.g., `node:20-alpine`, `openjdk:17`, `python:3.12`).
  - Volumes mounting each repo directory into its container.
  - A shared Docker network connecting all services.
  - Environment variables populated from decrypted workspace env keys (story 3.4).
  - Port mapping for each service's default port.
- [ ] **Backing services**: the DevOps agent detects stateful dependencies the app needs to boot (Postgres, MySQL, Redis, etc.) from the tech-stack metadata / config files and adds them as compose services — an app cannot be "run to test" without its database. These become the `backing_services` of the boot recipe (Story 9.6).
- [ ] **Monorepo-in-one-repo**: when the scanner (3.3) reports multiple runnable sub-packages inside a single repo (e.g. a pnpm/turbo monorepo with a frontend and a backend package), the generator emits **multiple services from that one repo directory**, not one — the "one service per repo" default must not collapse a monorepo into a single container.
- [ ] **Boot order & health**: services declare `depends_on` with `condition: service_healthy` and each service gets a `healthcheck` so a frontend does not start calling a backend that is not yet ready. Inter-service URLs are provided via the shared network + env, not hardcoded.
- [ ] The generated topology (services, backing services, ports, depends_on, healthchecks) is written as the AI-generated **boot recipe** (Story 9.6), `source: ai_generated`, `status: unvalidated` until a successful boot validates it.
- [ ] Generated compose file stored path recorded in DB (`workspaces.compose_file_path`).
- [ ] Preview endpoint: `GET /api/workspaces/:id/sandbox/compose-preview` — returns the compose file content as text (WORKSPACE_OWNER+).
- [ ] Migration: add `compose_file_path TEXT` column to `workspaces` table. Add migration file `packages/backend/src/db/migrations/add-workspace-compose-path.ts`.

## Technical Notes

- The LLM generates the compose file from a templated prompt that includes: repo names, detected stacks, env key names.
- Post-generation: validate the YAML is parseable before storing. Reject malformed output.
- Env values injected at runtime (not baked into compose file).
