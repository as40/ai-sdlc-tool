## Description

LLM-generated code routinely introduces injection flaws, hardcoded secrets, and vulnerable or wrongly-licensed dependencies. For an enterprise buyer, "what scans the AI's output before it can be committed?" is a day-one security-review question. This story runs three classes of scanner against the AI's changes inside the sandbox and blocks the workflow on critical findings.

## Scanner Classes

| Class                      | Tool (pluggable)                  | Catches                                                                     |
| -------------------------- | --------------------------------- | --------------------------------------------------------------------------- |
| SAST                       | Semgrep (default ruleset + OWASP) | Injection, unsafe deserialization, path traversal, weak crypto              |
| Secret detection           | Gitleaks / trufflehog-class       | Hardcoded API keys, tokens, private keys, connection strings                |
| SCA (dependency + license) | Trivy / osv-scanner-class         | Known CVEs in dependencies, disallowed licenses (GPL in a proprietary repo) |

Scanners are pluggable adapters behind a common interface so a workspace can swap in its enterprise-standard tools.

## Acceptance Criteria

### Scan Execution

- [ ] `SecurityScanService.scan(workspaceId, changedFiles): Promise<ScanReport>` runs the three scanner classes against the AI's changed files (diff scope, not the whole repo, for speed) inside the sandbox.
- [ ] Scanners run as ephemeral sidecar containers (or sandbox exec) — never on the host. No scanner is given network access beyond its own vulnerability DB mirror.
- [ ] `ScanReport` shape:
  ```typescript
  {
    findings: Array<{
      scanner: 'sast' | 'secret' | 'sca';
      severity: 'critical' | 'high' | 'medium' | 'low' | 'info';
      rule_id: string;
      file: string;
      line: number | null;
      message: string;
      remediation: string | null;
    }>;
    summary: {
      critical: number;
      high: number;
      medium: number;
      low: number;
    }
    passed: boolean;
  }
  ```

### Gate Policy

- [ ] The scan is invoked by the `VERIFYING` gate (Story 5.10) after tests/lint/typecheck pass, before advancing.
- [ ] Blocking policy (configurable per workspace, sane defaults):
  - Any `critical` finding → **block**. Feed findings back to the agent: `"Security scan found critical issues. Fix these before continuing:\n{findings}"`, transition `VERIFYING → RUNNING`.
  - Any hardcoded secret (`secret` scanner, any severity) → **always block** regardless of other policy. Secrets are non-negotiable.
  - `high` findings → block by default; workspace owner may downgrade to warn.
  - `medium`/`low`/`info` → warn only (surface to the human at approval, do not block).
- [ ] Secret findings are **redacted** in all output — the report shows the rule/file/line and a masked value (`sk-...abcd`), never the full secret, and the full secret is never logged or persisted.
- [ ] Self-correction on security findings counts against the Story 5.10 `MAX_VERIFY_RETRIES` budget and the workflow token/iteration budgets.

### Persistence & Reporting

- [ ] Every scan (pass or fail) is written to a `security_scan_reports` table: `id, workspace_id, workflow_execution_id, summary JSONB, findings JSONB (secrets masked), passed BOOLEAN, created_at`.
- [ ] `GET /api/workspaces/:id/security-scans` (WORKSPACE_OWNER+) — paginated scan history for compliance evidence.
- [ ] The approval payload (Story 5.5 `awaiting_approval` WS event) includes the latest `ScanReport` summary so the human sees the security posture of the change they are approving.

### Configuration

- [ ] `GET`/`PUT /api/workspaces/:id/security-policy` (WORKSPACE_OWNER+): configure blocking thresholds per scanner class, disallowed license list, and scanner tool selection.
- [ ] Scanners can be individually enabled/disabled, but the secret scanner cannot be disabled (enterprise safety floor) — attempting to disable it returns 400.

## Technical Notes

- Diff-scoped scanning keeps latency low; a full-repo scan option exists for the initial workspace onboarding (Phase 3) but is out of scope here.
- Scanner adapters normalise wildly different output formats (SARIF, JSON, custom) into the unified `ScanReport`. Add a SARIF parser since most tools can emit it.
- Vulnerability DBs must be mirrorable for air-gapped deployments (Phase 11) — do not hardcode calls to public scanner APIs.

## Tests to Write

- Clean change → scan passes, workflow advances.
- Critical SAST finding → blocked, findings fed back, status → `RUNNING`.
- Hardcoded secret at any severity → blocked even if policy would otherwise warn.
- Secret value masked in report and never logged (assert full value absent from persisted row and logs).
- High finding with owner downgrade policy → warns, does not block.
- Medium/low findings → surfaced in approval payload, not blocking.
- Disallowed license in a dependency → SCA blocks.
- Attempt to disable secret scanner → 400.
- Scan report persisted for pass and fail.
- SARIF parser normalises tool output to `ScanReport`.

## Definition of Done

- AI-generated changes are scanned for SAST, secrets, and vulnerable/mislicensed dependencies before reaching a human or completion. Critical issues and any secret block and feed back. Secrets never leak into logs or reports. History persisted for compliance. Tests pass.
