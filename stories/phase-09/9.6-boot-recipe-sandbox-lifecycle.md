## Description

Arbitrary projects cannot be booted reliably by automation alone. This story adds a **boot recipe** (a learned-once, structured description of how to bring a workspace up) and a **sandbox lifecycle state machine** with a graceful degradation ladder: the AI attempts a boot; if it fails, the human fixes it in the sandbox and the working setup is **captured into the recipe** so the system never has to guess again. Booting is always explicit and optional, and a running sandbox is never rebooted implicitly.

## Boot Recipe

- [ ] `boot_recipes` table:

  | Column                  | Type                                                                            |
  | ----------------------- | ------------------------------------------------------------------------------- |
  | id                      | UUID PK                                                                         |
  | workspace_id            | UUID FK → workspaces.id ON DELETE CASCADE                                       |
  | repo_id                 | UUID FK → repositories.id NULL (NULL = workspace-level / multi-repo)            |
  | service_name            | VARCHAR(200)                                                                    |

  | start_command           | TEXT NOT NULL                                                                   |

- [ ] `BootRecipeRepository`: `upsert()`, `getByWorkspace()`, `getValidated(workspaceId)`, `markValidated(id)`, `markFailed(id)`.
- [ ] A recipe becomes `validated` only after a boot using it reaches a healthy `RUNNING` state.

## Sandbox Lifecycle State Machine

- [ ] Sandbox status persisted on `workspaces` (or a `sandbox_state` table): `STOPPED | PREFLIGHT | AWAITING_BOOT_DECISION | BOOTING | RUNNING | FAILED | MANUAL_SETUP | HIBERNATED`.
- [ ] Transitions:
  - `STOPPED → PREFLIGHT` (user clicks **Boot** — never automatic).
  - `PREFLIGHT → BOOTING` if a `validated` recipe exists.
  - `PREFLIGHT → AWAITING_BOOT_DECISION` if no validated recipe exists (fires the HITL question below).
  - `BOOTING → RUNNING` when start commands complete and health checks pass → recipe marked `validated`.
  - `BOOTING → FAILED → MANUAL_SETUP` on any failure (logs surfaced to the terminal, Story 9.3).
  - `MANUAL_SETUP → RUNNING` when the human captures a working setup (see capture below).
  - `RUNNING → HIBERNATED` via Story 9.4; `HIBERNATED → RUNNING` on resume (uses the validated recipe, no re-decision).
  - `RUNNING → STOPPED` only on explicit stop or explicit restart.

## Optional, Explicit Boot (AI asks first)

- [ ] Boot is only ever triggered by an explicit user action (button or chat command) — the system never boots on login, file save, or AI code changes.
- [ ] On `AWAITING_BOOT_DECISION`, the orchestrator emits a `request_human_input` (Story 5.7) with options: `["Yes — let the AI try to boot it", "I'll set it up manually", "No boot needed"]` and a summary of the detected stack (from Story 3.3).
  - "Yes" → AI generates a recipe (Story 9.1) → `BOOTING`.
  - "I'll set it up manually" → `MANUAL_SETUP` directly (no AI attempt).
  - "No boot needed" → back to `STOPPED`.

## Manual Setup & Capture (the degradation ladder)

- [ ] In `MANUAL_SETUP`, the human uses the code editor (Phase 6), the live terminal (Story 9.3), and direct access to the workspace data volume to get the project running.
- [ ] Three capture mechanisms produce/​update a `boot_recipes` row (`source: human_*`, `status: validated`):
  1. **Terminal capture** (default): a **"Save these as boot steps"** action on the terminal (Story 9.3) lets the human select the commands they ran; those become `setup_commands` / `start_command`.
  2. **Chat instruction**: the human describes the setup in chat ("install `pnpm i`, migrate `pnpm db:migrate`, start `pnpm dev`, backend 3000 / frontend 5173"); a `BootRecipeExtractor` (economy-tier LLM, same pattern as Story 12.11) parses it into the structured recipe.
  3. **Recipe form**: a structured panel to enter commands, ports, order, and backing services explicitly.
- [ ] "I already set it up — just start it" shortcut: the human supplies only a start command; a minimal validated recipe is written and the sandbox boots directly, skipping AI generation.

## No Implicit Reboot Guard

- [ ] `POST .../sandbox/start` is **state-guarded**: if status is `RUNNING`, `BOOTING`, or `PREFLIGHT`, it returns `409 Conflict` (`SANDBOX_ALREADY_ACTIVE`) and does not restart.
- [ ] Restart is a **separate explicit endpoint** `POST .../sandbox/restart` requiring `{ confirm: true }`; it stops then boots using the validated recipe. Without `confirm`, returns 400.
- [ ] AI code changes, file saves, and new workflows never trigger a reboot. If the running app has `supports_hot_reload`, the UI does not even surface a restart prompt for ordinary edits.

## Tests to Write

- Boot with no validated recipe → status `AWAITING_BOOT_DECISION`, HITL question emitted.
- "No boot needed" → returns to `STOPPED`, nothing started.
- AI boot succeeds + health check passes → `RUNNING`, recipe marked `validated`.
- AI boot fails → `MANUAL_SETUP`, logs surfaced.
- Terminal capture → recipe saved with selected commands, `source: human_edited`, status `validated`.
- Chat instruction → extractor produces structured recipe; malformed input → warning, no crash.
- "Just start it" shortcut → minimal validated recipe, boots directly.
- `start` while `RUNNING` → 409, no reboot.
- `restart` without `confirm` → 400; with `confirm: true` → stop then boot.
- Resume from `HIBERNATED` → uses validated recipe, no boot-decision prompt.

## Definition of Done

- Boot is explicit and optional. AI attempts, falls back to human, and the human's working setup is captured into a reusable validated recipe. Running sandboxes never reboot implicitly; restart is explicit and confirmed. Tests pass.
