## Status

`[ ] PENDING`

## Description

The AI's context today comes from code embeddings (Phase 4) and Jira (Phase 7). But an enterprise's _design intent_ — the "why" behind the code — lives in Confluence, Notion, Google Docs, ADRs, runbooks, and internal wikis. An agent that can read these produces materially better, more on-intent output than one working from code alone. This story adds pluggable read-only knowledge connectors that ingest, chunk, embed, and make this material searchable alongside code, with strict access controls.

## Knowledge Sources (pluggable connectors)

| Connector              | Content                                                       |
| ---------------------- | ------------------------------------------------------------- |
| Confluence             | Spaces/pages — architecture docs, decisions, standards        |
| Notion                 | Databases/pages                                               |
| Google Drive / Docs    | Design docs, specs                                            |
| Markdown/ADR (in-repo) | `docs/**`, `**/adr/**`, `README`s already in the cloned repos |
| Generic web/URL        | Allow-listed internal doc URLs                                |

## Acceptance Criteria

### Connectors

- [ ] `KnowledgeConnector` interface: `authenticate(config)`, `listDocuments()`, `fetchDocument(id)`, all read-only. Connectors are adapters so new sources can be added without touching the ingestion core.
- [ ] Connector credentials stored encrypted (Story 1.3 CryptoService), same pattern as AI provider keys.
- [ ] At least the Confluence and in-repo Markdown/ADR connectors implemented in this story; others are stubs conforming to the interface.
- [ ] `knowledge_sources` table: `id, workspace_id, connector_type, config_encrypted, sync_status, last_synced_at, doc_count, enabled, created_at`.

### Ingestion

- [ ] `POST /api/workspaces/:id/knowledge-sources/:sourceId/sync` (WORKSPACE_OWNER+) — pulls documents, chunks them (reuse Story 4.2 chunker with a prose-aware strategy, not AST), embeds, and upserts into the vector store with a `source_type='knowledge'` discriminator and `source_ref` (URL/id) metadata.
- [ ] Incremental sync: unchanged documents (by content hash) are skipped, mirroring the code-embedding `file_hash` optimisation.
- [ ] Sync runs as a background job with WS progress events; large sources do not block the request.
- [ ] Deleting/disabling a source removes its embeddings from the vector store.

### Retrieval & Access Control

- [ ] `search_codebase` is generalised (or a sibling `search_knowledge` skill added) so the orchestrator's RAG step (Story 5.4) retrieves from **both** code and knowledge embeddings, with results tagged by `source_type` so the agent knows whether it is reading code or documentation.
- [ ] Retrieved knowledge chunks carry their `source_ref` so the agent (and the human) can cite where guidance came from.
- [ ] **Access control**: knowledge embeddings are workspace-scoped; a sync only ingests documents the connector's configured service account can access. Ingested content is subject to the Story 5.11 DLP layer when later sent to a cloud provider.
- [ ] Per-source enable/disable and a workspace toggle to exclude knowledge from RAG entirely (some teams want code-only context).

## Technical Notes

- Prose chunking differs from code chunking — chunk on headings/paragraphs with overlap, not AST boundaries. Keep chunk size aligned with the embedding model's optimal window.
- Respect source rate limits and pagination; syncs must be resumable.
- This is strictly read-only ingestion — no connector writes back to Confluence/Notion/etc. State that explicitly to avoid scope creep.
- Reuse the existing `code_embeddings` table with a discriminator column if simplest, or a parallel `knowledge_embeddings` table sharing the vector index approach from Story 4.1 — pick one and note it; do not duplicate the embedding infrastructure.
