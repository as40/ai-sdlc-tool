# Story 8.3 — Pull Request Automation (Post-Approval)

## Phase

Phase 8: Automated Branching & Version Control Operations

## Dependencies

- Story 5.5, 8.1 must be `DONE`.

## Status

`[ ] PENDING`

## Description

Build the post-COMPLETED execution pipeline: stage all modified files, commit with an AI-generated message, push to remote, and open a Pull Request.

## Acceptance Criteria

- [ ] `PRAutomationService` triggered automatically when workflow transitions to `COMPLETED`.
- [ ] Pipeline steps:
  1. `git add -A` — stage all modified working-directory files.
  2. Call LLM to generate a commit message summarizing the changes (based on git diff).
  3. `git commit -m "<ai-generated message>"`.
  4. `git push origin feature/JIRA-XXX`.
  5. Create a PR via the Git provider API (GitHub: `POST /repos/{owner}/{repo}/pulls`).
- [ ] PR title: Jira story key + summary (e.g., `[PROJ-42] Add user authentication`).
- [ ] PR body: AI-generated description including: what changed, why (from Jira story), test notes.
- [ ] PR URL stored in `workflow_executions` table (`pr_url TEXT`).
- [ ] WebSocket event: `{ type: 'pr_created', payload: { prUrl, branch } }`.
- [ ] If push fails (e.g., remote conflicts): transition workflow to `REVISION_REQUESTED` with error context.
- [ ] Migration: add `pr_url TEXT` column to `workflow_executions` table. Add migration file `packages/backend/src/db/migrations/add-workflow-pr-url.ts`.
- [ ] If no AI config is set for the workspace, commit message generation falls back to a template: `'feat: AI-generated changes for workflow {workflowExecutionId}'` rather than calling the LLM.
