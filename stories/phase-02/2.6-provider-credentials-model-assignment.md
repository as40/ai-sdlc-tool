## Phase

Phase 2: Authentication & Secure Configuration Setup

- Story 1.3 (CryptoService) must be `DONE`.

## Status

`[ ] PENDING`

## Description

Story 2.5 modeled AI config as one `{ provider + model + key }` row, which forces re-entering the same key per model and hides the fact that **one provider key serves all of that provider's models**. This story restructures it to match how provider keys actually work: **one credential → many model assignments**, each mapped to a tier. It also adds the scope hierarchy enterprises need (org → workspace → user), provider-aware model discovery (including local models), and a strict "store the exact model string" rule so the value passed to the provider at task time is always correct.

This story supersedes 2.5's single-table model via migration; 2.5's endpoints are re-pointed at the new structure (or kept as compatibility shims).

## Product Expectation: BYOK for every customer size

The software is **Bring-Your-Own-Key first** and provider-neutral — any customer runs it with their own Claude, Gemini, OpenAI, or local model. The scope hierarchy below is what lets the _same_ software serve every customer tier with no code difference; only the scope of the credential changes:

| Customer tier           | Where the key lives                                                                                                                    | Scope                |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | -------------------- |
| Enterprise              | One central provider key, or their AWS Bedrock / GCP Vertex / Azure account, managed by an org admin and shared across all their teams | `org`                |
| Mid-level               | One team key configured by the workspace owner, shared within that workspace                                                           | `workspace`          |
| Small team / individual | A single person's key for their workspace, or their own user override                                                                  | `workspace` / `user` |

- One provider key serves all of that provider's models (e.g. a single Gemini key → Flash for economy, Pro for heavy work; a single Anthropic key → Haiku/Sonnet/Opus).
- Provider billing goes directly to the customer (it's their key); the platform only meters usage for the customer's own reporting (Stories 10.1 / 10.5).
- **Minimum requirement:** at least one credential must exist at some scope, or the gateway returns `NO_AI_CONFIG` and no AI task can run — a model is the one non-optional integration. A mid-level customer satisfies this with a single workspace-level key.
- A future **platform-managed** option (for customers with no provider account) is simply an `org`-scoped credential the platform provides — no separate code path.

## Data Model

- [ ] `provider_credentials` table — the key, entered once per provider:

  | Column                  | Type                                                                                            |
  | ----------------------- | ----------------------------------------------------------------------------------------------- |
  | id                      | UUID PK                                                                                         |
  | scope_type              | VARCHAR(20) CHECK IN ('org','workspace','user')                                                 |
  | scope_id                | UUID (org id / workspace id / user id)                                                          |
  | provider                | VARCHAR(40) CHECK IN ('anthropic','openai','gemini','bedrock','vertex','azure_openai','custom') |
  | api_key_encrypted       | TEXT NULL (NULL allowed for cloud-IAM providers like bedrock/vertex)                            |
  | base_url                | TEXT NULL (Azure/Bedrock region endpoint, or local/self-hosted URL)                             |
  | extra_config_encrypted  | JSONB NULL (e.g. AWS region/role, Azure api-version)                                            |
  | is_local                | BOOLEAN DEFAULT false                                                                           |
  | created_at / updated_at | TIMESTAMPTZ                                                                                     |

- [ ] `model_assignments` table — which models from that credential, and what role each plays:

  | Column                  | Type                                                                                                                             |
  | ----------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
  | id                      | UUID PK                                                                                                                          |
  | credential_id           | UUID FK → provider_credentials.id ON DELETE CASCADE                                                                              |
  | tier                    | VARCHAR(20) CHECK IN ('default','economy','standard','premium')                                                                  |
  | model_id                | TEXT NOT NULL — the **exact** string the provider expects (e.g. `claude-opus-4-8`, `llama3.1:8b`, or an Azure _deployment name_) |
  | purpose                 | VARCHAR(20) CHECK IN ('chat','embedding') DEFAULT 'chat'                                                                         |
  | display_name            | TEXT NULL (friendly label for the UI)                                                                                            |
  | validated_at            | TIMESTAMPTZ NULL (set when a test call last succeeded)                                                                           |
  | created_at / updated_at | TIMESTAMPTZ                                                                                                                      |

- [ ] Unique constraint on `(scope_type, scope_id, provider)` for credentials, and `(credential_id, tier, purpose)` for assignments (one model per tier+purpose per credential).
- [ ] `api_key_encrypted` / `extra_config_encrypted` encrypted via CryptoService; never returned in plaintext (masked on GET).

## Scope Resolution Hierarchy

- [ ] Keys are **shared, not per-user by default.** Resolution is most-specific-wins: **user → workspace → org**. The gateway (Story 5.4) resolves a tier to a concrete `{ credential, model_id }` by walking this chain.
- [ ] Org-level credentials are the enterprise default (one central contract / cloud account). Workspace-level = cost-center BYOK. User-level = rare personal/compliance override.
- [ ] Per-user **attribution** is independent of key ownership — every call is logged with `user_id` (Story 10.1), so a shared key still yields per-user usage reporting.
- [ ] A platform-managed offering is simply an org-scoped credential the platform provides — no separate code path (BYOK and managed share this model).

## Model Discovery (provider-aware dropdown)

- [ ] `GET /api/.../credentials/:id/available-models` returns the selectable models for a credential, resolved by provider:
  - Cloud with a list API (anthropic, openai, gemini) → **live fetch** using the stored key (`GET /v1/models` or equivalent).
  - `bedrock` / `vertex` → list via the cloud SDK using the stored IAM/region config.
  - `azure_openai` → list the account's **deployments** (Azure calls the _deployment name_, not the base model id).
  - `is_local` (Ollama / vLLM / LM Studio / OpenAI-compatible) → query the endpoint (`GET /api/tags` for Ollama, `GET /v1/models` for OpenAI-compatible) at `base_url`.
  - On fetch failure or unknown provider → return an empty list and let the UI fall back to **free-text entry**.
- [ ] The form model field is a **combo box**: dropdown populated from the discovery call, plus always-available free-text entry for a model not yet listed. Whatever is chosen/typed is stored verbatim as `model_id`.

## Exact-Value Guarantee & Validation

- [ ] The stored `model_id` is passed **unchanged** as the provider's `model` parameter at task time (Story 5.4 adapters). No normalization, aliasing, or case-changing — the platform never assumes a canonical id.
- [ ] `POST /api/.../model-assignments/:id/test` performs a minimal (1-token) call using the exact `credential` + `model_id`. Success sets `validated_at`; failure returns the provider error so a bad model string is caught in settings, not mid-workflow.
- [ ] An unvalidated assignment can be saved but is flagged `unvalidated` in the UI; the gateway logs a warning if it must use an unvalidated assignment.

## API & Form

- [ ] CRUD endpoints for credentials and model-assignments, scoped by RBAC: WORKSPACE_OWNER manages workspace/user scope within their workspace; an ORG/SUPER_ADMIN manages org scope. Keys never returned in plaintext.
- [ ] Form flow (supersedes the 2.5 single form):
  1. **Add credential** — provider dropdown, key (masked), base URL / cloud config for bedrock/vertex/azure/local, `is_local` toggle, **Test** button.
  2. **Assign models** — for that credential, add rows: tier → model (combo box from discovery) → purpose, each with a **Test** button.
- [ ] All inputs use shadcn/ui components.

## Migration from 2.5

- [ ] Migration converts each existing `ai_configurations` row into one `provider_credentials` (deduped per provider+scope) plus a `model_assignments` row carrying its `model_id` and existing `tier`.
- [ ] 2.5's `GET/POST/PUT/DELETE /ai-config` endpoints are re-pointed to the new tables or kept as thin compatibility wrappers so no consumer breaks.

## Tests to Write

- One credential + three assignments (opus→premium, sonnet→standard, haiku→economy) share the same key.
- Anthropic discovery → live model list returned (mock provider API).
- Local (Ollama) discovery → models from `GET /api/tags` at base_url.
- Discovery failure / unknown provider → empty list, free-text still accepted.
- `model_id` stored and returned verbatim (e.g. Azure deployment name preserved, no normalization).
- Test call success → `validated_at` set; failure → provider error surfaced, not saved as validated.
- Resolution: user-scope assignment overrides workspace overrides org (most-specific wins).
- No credential at any scope → resolution returns `NO_AI_CONFIG`.
- Key encrypted at rest; masked on GET; never logged.
- 2.5 migration → existing configs become credential + assignment without data loss.
