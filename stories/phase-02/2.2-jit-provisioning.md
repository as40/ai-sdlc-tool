# Story 2.2 — Just-in-Time (JIT) Auto-Provisioning

## Phase
Phase 2: Authentication & Secure Configuration Setup

## Dependencies
- Story 2.1 must be `DONE`.

## Status
`[ ] PENDING`

## Description
Create the JIT provisioning middleware that intercepts successful SSO logins and automatically creates a new user record if one does not exist, then issues a JWT for the React UI.

## Acceptance Criteria
- [ ] `JitProvisioningMiddleware` is invoked after every successful SSO callback (OIDC and SAML).
- [ ] If the user's email **does not exist** in the `users` table: insert a new record with `access_level = 'DEVELOPER'` (default).
- [ ] If the user's email **already exists**: update `display_name` from the SSO profile if it differs.
- [ ] After provisioning, issue a signed JWT (same format as mock login) and redirect to the frontend with the token as a query param or via a secure cookie.
- [ ] The `display_name` is populated from the SSO profile's `name` or `given_name + family_name` fields.
- [ ] `deleted_at` check: if the user exists but has `deleted_at` set, reject login with a `403` and an error page.
- [ ] All DB operations are wrapped in a transaction — partial provisioning does not leave orphaned records.
- [ ] Middleware logs the provisioning event (new vs. existing) using the structured logger.

## Technical Notes
- Middleware signature: `(profile: SsoProfile, done: (err, user) => void) => Promise<void>`.
- `SsoProfile`: `{ email: string; displayName: string; providerId: string }`.
- Use the `UserRepository` for DB access (follow layered architecture).
- After provisioning, call the existing JWT signing utility from story 2.0.

## Tests to Write
### `jit-provisioning.test.ts`
- New email: creates user, returns JWT.
- Existing email: does NOT create duplicate, returns JWT.
- Existing email with updated display_name: updates the record.
- Soft-deleted user: returns 403, does not issue token.
- DB failure mid-transaction: rolls back and returns 500.

## Definition of Done
- All tests pass.
- New user is visible in the `users` table after first SSO login.
- No duplicate users created on repeated logins with the same email.
