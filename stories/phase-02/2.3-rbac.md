# Story 2.3 — Role-Based Access Control (RBAC)

## Phase

Phase 2: Authentication & Secure Configuration Setup

## Dependencies

- Story 2.0 must be `DONE`.

## Status

`[ ] PENDING`

## Description

Implement the RBAC middleware that enforces role-based restrictions on backend routes. Ensure `DEVELOPER` and `VIEWER` accounts cannot access workspace settings or AI key configurations.

## Acceptance Criteria

- [ ] `requireRole(...roles: AccessLevel[])` middleware factory created.
- [ ] If the authenticated user's role is NOT in the allowed roles array: return `403 Forbidden` with RFC 7807 error body.
- [ ] Role hierarchy understood by the system:
  - `SUPER_ADMIN` > `WORKSPACE_OWNER` > `DEVELOPER` > `VIEWER`
  - `SUPER_ADMIN` can access everything.
- [ ] The following routes are protected:
  - Workspace settings (`/api/workspaces/:id/settings`): `WORKSPACE_OWNER`, `SUPER_ADMIN` only.
  - AI config (`/api/workspaces/:id/ai-config`): `WORKSPACE_OWNER`, `SUPER_ADMIN` only.
  - SSO settings (`/api/admin/sso`): `SUPER_ADMIN` only.
  - User management (`/api/admin/users`): `SUPER_ADMIN` only.
- [ ] `DEVELOPER` calling a `WORKSPACE_OWNER`-only route returns `403`.
- [ ] `VIEWER` calling a `DEVELOPER`-level route (if any) returns `403`.
- [ ] Frontend: Role is read from the decoded JWT. Protected UI routes use a `<ProtectedRoute role="WORKSPACE_OWNER">` wrapper component that redirects to `/unauthorized` if the role is insufficient.
- [ ] `UnauthorizedPage.tsx` renders at `/unauthorized` — full-screen message "You don't have permission to view this page" with a "Go back" button. This route requires NO `ProtectedRoute` wrapper (publicly accessible).
- [ ] On app refresh, if `auth_token` exists in `localStorage`, the Zustand auth store re-hydrates the token and decodes the JWT payload to restore `user` state (`role`, `sub`, `email`) without a network call. If the token is expired or invalid, `clearToken()` is called.

## Technical Notes

- Middleware reads `req.user.role` set by `requireAuth`.
- `requireRole` must be chained AFTER `requireAuth` on every protected route.
- Frontend `ProtectedRoute` reads from the auth Zustand store (decoded JWT).
- `AccessLevel` enum defined once in a shared types file, imported by both middleware and schema.
