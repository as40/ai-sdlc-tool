# Story 13.8 — Agent Evaluation Harness & Regression Detection

## Phase

Phase 13: Advanced Workspace Customization & Validation Guardrails

## Dependencies

- Story 5.4 must be `DONE` (workflow execution to run eval tasks through).
- Story 13.5 must be `DONE` (config inheritance — evals run against a specific resolved config).
- Story 5.10 should be `DONE` (verification results feed the scoring signal).

## Status

`[ ] PENDING`

## Description

Phase 13 lets workspace owners customise prompts, personas, and agent configs — but nothing tells them whether a change made the AI _better or worse_. Without measurement, a well-intentioned prompt tweak can silently degrade quality across the whole workspace. This story adds a golden-task evaluation harness: a fixed set of representative tasks with expected outcomes that the AI is scored against on every meaningful config change, giving owners a quality signal and catching regressions before they ship. It is also the ROI/quality story an enterprise buyer wants: "the AI passes 94% of golden tasks."

## Acceptance Criteria

### Eval Sets

- [ ] `eval_tasks` table: `id, workspace_id, name, prompt TEXT, fixture_repo_ref TEXT, assertions JSONB, tags TEXT[], enabled BOOLEAN, created_at`.
- [ ] An `assertion` supports multiple checkable types (composable per task):
  - `files_changed`: expected file paths (or globs) that should be modified.
  - `must_contain` / `must_not_contain`: substrings/regex expected (or forbidden) in the diff.
  - `verification_passes`: the Story 5.10 checks (tests/lint/typecheck) must pass.
  - `no_critical_security_findings`: the Story 9.5 scan must be clean of criticals.
  - `llm_judge`: a rubric evaluated by an LLM-as-judge call (economy tier), returning a 0–100 score against a described rubric — for qualitative aspects that exact matching can't capture.
- [ ] `GET`/`POST`/`PUT`/`DELETE /api/workspaces/:id/eval-tasks` (WORKSPACE_OWNER+) to manage the eval set. Seed a small default set for common stacks.

### Running Evals

- [ ] `POST /api/workspaces/:id/eval-runs` (WORKSPACE_OWNER+): runs all enabled eval tasks against the workspace's current resolved config, each in an isolated sandbox against its fixture, in parallel (bounded concurrency).
- [ ] Each task produces a result: `{ taskId, passed, score, assertion_results: [...], tokens_used, latency_ms, workflow_execution_id }`.
- [ ] An `eval_runs` record aggregates: `id, workspace_id, config_snapshot JSONB, pass_rate NUMERIC, avg_score NUMERIC, total_tokens, triggered_by, created_at`, plus per-task results in `eval_run_results`.
- [ ] Runs are executed as background jobs (do not block the request); progress streamed via WS `{ type: 'eval_progress', payload: { runId, completed, total } }`.

### Regression Detection

- [ ] When a workspace owner saves an agent/prompt/config change (integrates with Story 13.2/13.4 save path), the system can trigger an eval run against the **proposed** config and compare to the last baseline run:
  - If `pass_rate` drops by more than `EVAL_REGRESSION_THRESHOLD` (default 10 percentage points) or any previously-passing task now fails, surface a **regression warning** with the specific tasks that regressed.
  - The owner sees a diff: "3 tasks regressed, 1 improved" with per-task detail, and can choose to proceed or roll back the config change.
- [ ] `GET /api/workspaces/:id/eval-runs` and `.../eval-runs/:id` — history and detail, so owners can chart quality over time.

### Reporting

- [ ] Eval results feed a quality summary surfaced on the analytics dashboard (Story 10.3): current pass rate, trend, and reviewer-catch-rate (from Story 5.12 history) as complementary quality signals.

## Technical Notes

- Golden tasks must be deterministic enough to be meaningful but the AI is stochastic — use pass/fail assertions for hard checks and the `llm_judge` score (averaged, with a fixed judge prompt and low temperature) for soft quality. Report both; don't over-index on a single stochastic score.
- Eval runs consume real tokens against the workspace's provider — respect the Story 10.4 rate limits and Story 10.5 budgets; a full eval run must be budget-aware and abortable.
- Keep fixtures small and self-contained (tiny repos) so runs are fast and cheap. Store fixtures in workspace file storage (Story 14.3) or a dedicated eval-fixtures area.
- LLM-judge is itself subject to the DLP layer (Story 5.11) if it runs against a cloud provider.

## Tests to Write

- Eval task with `files_changed` assertion → passes when expected files modified, fails otherwise.
- `must_not_contain` assertion → fails when forbidden pattern present in diff.
- `verification_passes` assertion → reflects Story 5.10 result.
- `llm_judge` assertion → returns a bounded 0–100 score; deterministic-ish with fixed judge prompt + low temperature (mock the judge in unit tests).
- Full eval run → aggregate pass_rate and avg_score computed correctly.
- Proposed config drops pass rate beyond threshold → regression warning lists the regressed tasks.
- Previously-passing task now failing → flagged even if overall pass rate barely moves.
- Eval run respects workspace budget/rate limits (aborts when exceeded).
- Eval progress WS events emitted.

## Definition of Done

- Workspace owners can measure AI quality against a golden task set and catch regressions from config changes before shipping them. Runs are sandboxed, budget-aware, and backgrounded. Quality trend surfaced on the analytics dashboard. Tests pass.
