# Story 13.7 — Policy-Based Approval Routing

## Phase

Phase 13: Advanced Workspace Customization & Validation Guardrails

## Dependencies

- Story 5.5 must be `DONE` (base single-approver HITL flow that this story generalises).
- Story 2.3 must be `DONE` (RBAC — approval policies reference roles).

## Status

`[ ] PENDING`

## Description

Enterprise change management is not "any developer clicks approve." Sensitive changes need the right approvers: schema or auth changes may require a senior/owner sign-off; production config changes may require two approvers. This story adds a policy engine on top of the existing HITL approval gate that routes each `AWAITING_APPROVAL` to the correct required approver(s) based on **what the change touches**.

## Acceptance Criteria

### Policy Model

- [ ] `approval_policies` table: `id, workspace_id, name, priority INT, match JSONB, requirement JSONB, enabled BOOLEAN, created_at, updated_at`.
- [ ] `match` describes when the policy applies, evaluated against the change:
  - `path_globs`: file path patterns (e.g. `**/migrations/**`, `**/auth/**`, `**/*.tf`).
  - `security_severity_at_least`: applies if the Story 9.5 scan reported a finding at/above this severity.
  - `touches_infra`: boolean heuristic (Dockerfiles, IaC, CI config).
- [ ] `requirement` describes what approval is needed:
  - `min_approvals`: integer (e.g. 2).
  - `required_roles`: e.g. `['WORKSPACE_OWNER']` — at least one approval must come from a listed role.
  - `distinct_approvers`: boolean — the N approvals must be from different users (no self double-approval).
  - `forbid_author`: boolean — the workflow initiator cannot be a required approver.
- [ ] Policies evaluated in `priority` order; the **most specific matching policy wins** (highest priority). If no policy matches, the default requirement applies (single `DEVELOPER`+ approval, as today).

### Enforcement

- [ ] When a workflow enters `AWAITING_APPROVAL`, the engine computes the effective requirement from the change's diff + latest scan report and attaches it to the approval state: `{ required_approvals, required_roles, current_approvals: [] }`.
- [ ] `POST .../approve` records an approval from the calling user but does **not** transition to `RUNNING`/`COMPLETED` until the requirement is fully satisfied.
  - Records `{ userId, role, at }` in `current_approvals`.
  - Rejects a second approval from the same user when `distinct_approvers` is set (409).
  - Rejects approval by the author when `forbid_author` is set (403).
  - Returns `{ satisfied: boolean, remaining: { approvals, roles } }`.
- [ ] Only when the requirement is satisfied does the workflow advance. Emits `{ type: 'approval_satisfied', payload: { workflowId } }`.
- [ ] `POST .../revise` (reject) at any point discards accumulated approvals and sends the change back (existing 5.5 behaviour) — a rejection resets the approval tally.
- [ ] The approval state and every individual approval are recorded in `workflow_executions.history` for audit (who approved, which role, when).

### UI / API

- [ ] `GET`/`POST`/`PUT`/`DELETE /api/workspaces/:id/approval-policies` (WORKSPACE_OWNER+) to manage policies.
- [ ] The `awaiting_approval` WS payload includes the effective requirement and current progress so the IDE can show "1 of 2 approvals — needs a WORKSPACE_OWNER."
- [ ] Policy management UI in the Phase 13 configuration dashboard (Story 13.1 tabs).

## Technical Notes

- Keep matching deterministic and explainable — the approval panel should be able to show _why_ a given policy applied ("matched `**/migrations/**`"). Store the matched policy id/name on the approval state.
- This composes with Story 9.5: a security scan surfacing a `high` finding can itself trigger a stricter approval policy via `security_severity_at_least`.
- Do not trust client-sent roles — resolve the approver's role server-side from the session (consistent with the RBAC rules in CLAUDE.md).

## Tests to Write

- No policy matches → default single-approval behaviour (no regression vs 5.5).
- Change touching `**/migrations/**` → policy requires WORKSPACE_OWNER approval; DEVELOPER approval alone does not advance.
- `min_approvals: 2` + `distinct_approvers` → same user approving twice → 409, still not satisfied.
- `forbid_author` → workflow initiator approving → 403.
- Two distinct qualifying approvals → requirement satisfied, workflow advances, `approval_satisfied` emitted.
- Reject after one approval → approval tally reset.
- Highest-priority matching policy wins when multiple match.
- `security_severity_at_least` policy triggered by a high scan finding.
- Every approval recorded in history with user + role + timestamp.

## Definition of Done

- Approvals routed to the correct required approvers based on what the change touches. Multi-approval, distinct-approver, role, and no-self-approval rules enforced server-side. Full approval audit trail. Default behaviour unchanged when no policy matches. Tests pass.
