# Story 3.1 — Provider-Level Git Authentication

## Phase
Phase 3: Workspace Onboarding & Multi-Repo Universal Scanner

## Dependencies
- Story 2.3, 2.5 must be `DONE`.

## Status
`[ ] PENDING`

## Description
Build a Git provider authentication integration in Workspace Settings. Allow a `WORKSPACE_OWNER` to connect a GitHub or Bitbucket account using OAuth or a Personal Access Token (PAT). Store the token securely and link it to the workspace.

## Acceptance Criteria
- [ ] `git_provider_tokens` table: `id, workspace_id, provider ('github'|'bitbucket'), token_encrypted TEXT, scopes TEXT[], created_at, updated_at`.
- [ ] `POST /api/workspaces/:id/git-auth` — save a PAT (WORKSPACE_OWNER+ only). Encrypts token before storage.
- [ ] `GET /api/workspaces/:id/git-auth` — return provider name and masked token (`****` + last 4 chars). Never returns plaintext.
- [ ] `DELETE /api/workspaces/:id/git-auth` — revoke the stored token.
- [ ] `POST /api/workspaces/:id/git-auth/test` — validates the token by calling the provider API (`GET /user` on GitHub). Returns `{ valid: boolean; username: string; scopes: string[] }`.
- [ ] Frontend: Git settings section within Workspace Settings page with PAT input form and "Test Connection" button.
- [ ] Optional: GitHub OAuth flow (if implementing, use `passport-github2`). PAT is the primary path.

## Technical Notes
- Use `@octokit/rest` for GitHub API calls.
- Token is decrypted in memory only for API calls — never stored decrypted.
- Scope check: warn if the token is missing the `repo` scope (needed for cloning private repos).
