## Description

Build a Git provider authentication integration in Workspace Settings. Allow a `WORKSPACE_OWNER` to connect a GitHub or Bitbucket account using OAuth or a Personal Access Token (PAT). Store the token securely and link it to the workspace.

## Acceptance Criteria

- [ ] `git_provider_tokens` table: `id, workspace_id, provider ('github'|'bitbucket'), token_encrypted TEXT, scopes TEXT[], created_at, updated_at`.
- [ ] `POST /api/workspaces/:id/git-auth` ‚Äî save a PAT (WORKSPACE_OWNER+ only). Encrypts token before storage.
- [ ] `GET /api/workspaces/:id/git-auth` ‚Äî return provider name and masked token (`****` + last 4 chars). Never returns plaintext.
- [ ] `DELETE /api/workspaces/:id/git-auth` ‚Äî revoke the stored token.
- [ ] `POST /api/workspaces/:id/git-auth/test` ‚Äî validates the token by calling the provider API (`GET /user` on GitHub). Returns `{ valid: boolean; username: string; scopes: string[] }`.
- [ ] Frontend: Git settings section within Workspace Settings page with PAT input form and "Test Connection" button.
- [ ] Optional: GitHub OAuth flow (if implementing, use `passport-github2`). PAT is the primary path.
- [ ] Frontend Git settings panel lives at standalone route `/workspaces/:id/settings/git` in `App.tsx` until Story 13.1 migrates it into the Workspace Settings tab shell. Requires `ProtectedRoute` with `WORKSPACE_OWNER` role.
- [ ] PAT test connection button shows loading state (`'Testing‚Ä¶'`), success (`<Alert variant="success">` "Connection verified"), and error (`<Alert variant="error">` with message from `detail`).
- [ ] All form inputs use shadcn/ui `Input`, `Label`, `Button`, `Select` components.

## Technical Notes

- Use `@octokit/rest` for GitHub API calls.
- Token is decrypted in memory only for API calls ‚Äî never stored decrypted.
- Scope check: warn if the token is missing the `repo` scope (needed for cloning private repos).
