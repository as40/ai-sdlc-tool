# Story 3.2 — Multi-Repo Selection & Cloning Wizard

## Phase

Phase 3: Workspace Onboarding & Multi-Repo Universal Scanner

## Dependencies

- Story 3.1 must be `DONE`.

## Status

`[ ] PENDING`

## Description

Build an onboarding UI wizard that lists the user's available repositories (using the stored Git token), allows selecting a primary and multiple secondary repos, then clones them all into a shared parent workspace directory on the server.

## Acceptance Criteria

- [ ] `GET /api/workspaces/:id/repos/available` — fetches the user's repo list from the Git provider using the stored token. Returns `[{ name, remoteUrl, isPrivate, defaultBranch }]`.
- [ ] `POST /api/workspaces/:id/repos/clone` — begins cloning selected repos.
  - Request: `{ repos: [{ remoteUrl: string; isPrimary: boolean }] }`.
  - Validates: exactly one repo has `isPrimary: true`.
  - Clones repos into `WORKSPACE_BASE_PATH/{workspaceId}/{repoName}/` using `simple-git`.
  - Stores each repo in the `repositories` table with `local_path`.
  - Streams cloning progress via WebSocket events: `{ type: 'clone_progress', payload: { repoName, progress, status } }`.
- [ ] Cloning runs in a background worker (not blocking the HTTP response). Endpoint returns `202 Accepted` with a `jobId`.
- [ ] `GET /api/workspaces/:id/repos/clone-status/:jobId` — poll for job status.
- [ ] Frontend wizard: step 1 — list repos with checkboxes; step 2 — mark primary; step 3 — progress screen with HTTP polling (every 2 seconds) per repo (replaced by WS in Story 6.1).
- [ ] Parent workspace directory is created if it doesn't exist.
- [ ] Cloning with the stored PAT for private repos: `https://{token}@github.com/...`.
- [ ] Clone failure (non-200 status or `status: 'failed'` in poll response) shows `<Alert variant="error">` with the error message and a 'Try Again' button.
- [ ] On successful clone completion, wizard navigates to `/workspaces/:id/repos` (or the workspace overview once Story 4.3 defines it).
- [ ] All wizard UI uses shadcn/ui components.

## Technical Notes

- Use `simple-git` npm package for programmatic git operations.
- `WORKSPACE_BASE_PATH` from env var (e.g., `/data/workspaces`).
- Background worker: use Node.js `worker_threads` or a simple in-memory job queue for now (BullMQ in a later phase if needed).
- Security: sanitize repo names used in file paths. Reject names with `..` or special chars.
- **WebSocket client not yet available:** WebSocket client infrastructure is not established until Story 6.1. For Phase 3, the wizard's progress screen uses HTTP polling (`GET /api/workspaces/:id/repos/clone-status/:jobId` every 2 seconds) instead of WebSocket. When Story 6.1 is complete, the polling hook can be replaced with the WS event `clone_progress` without changing the component interface.
