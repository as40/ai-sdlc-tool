# Story 4.1 — Database Schema for Vector Memory

## Phase
Phase 4: Full Vectorization & Long-Term Memory Mapping

## Dependencies
- Story 1.2 (pgvector enabled) must be `DONE`.

## Status
`[ ] PENDING`

## Description
Create the `code_embeddings` table with the pgvector column type, indexes, and Drizzle ORM schema definition.

## Acceptance Criteria
- [ ] `code_embeddings` table created via migration with these columns:
  | Column | Type | Constraints |
  |--------|------|-------------|
  | id | UUID (v7) | PK |
  | repo_id | UUID | FK → repositories.id ON DELETE CASCADE |
  | file_path | TEXT | NOT NULL |
  | chunk_index | INTEGER | NOT NULL |
  | content | TEXT | NOT NULL |
  | file_hash | VARCHAR(64) | NOT NULL |
  | embedding | vector(768) | NOT NULL |
  | created_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() |
  | updated_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() |
- [ ] Unique constraint on `(repo_id, file_path, chunk_index)`.
- [ ] ivfflat index on `embedding` column for approximate nearest-neighbor search: `CREATE INDEX ON code_embeddings USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100)`.
- [ ] `EmbeddingRepository` class with:
  - `upsertChunk(data)` — insert or update on conflict of unique constraint.
  - `similaritySearch(queryVector, repoId, limit)` — returns top-N most similar chunks.
  - `deleteByRepo(repoId)` — bulk delete for re-indexing.
  - `deleteByFile(repoId, filePath)` — incremental delete for file-level re-vectorization.

## Technical Notes
- Vector dimension 768 is for `nomic-embed-text` / `text-embedding-3-small` compatible models. This can be overridden per workspace config later.
- Drizzle ORM does not have native vector type — use `customType` or raw SQL in migration.
- `file_hash` is SHA-256 of the file content, used to skip re-embedding unchanged files.
