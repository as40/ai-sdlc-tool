# Story 4.4 — Workspace AI Memory Store

## Phase

Phase 4: Full Vectorization & Long-Term Memory Mapping

## Dependencies

- Story 1.2 (database) must be `DONE`.
- Story 4.1 (vector schema) must be `DONE` — this story complements vector search with structured memory; they are separate systems serving different purposes.

## Status

`[ ] PENDING`

## Description

Create a persistent structured memory store where AI agents can save and recall learned facts across sessions. This is distinct from the vector codebase embeddings (Story 4.1), which index source code for similarity search. This store holds **what the AI has learned about the workspace** — coding patterns that work, team preferences inferred from human corrections, architectural constraints, and task checkpoints.

This is the foundation for all cross-session AI memory. Other stories (12.10, 12.11, 12.4) build on top of it.

## Memory Types

| Type                     | What it stores                                                           | Example                                                                       |
| ------------------------ | ------------------------------------------------------------------------ | ----------------------------------------------------------------------------- |
| `pattern`                | A repeatable approach the AI discovered that works in this workspace     | "API routes always import auth middleware from `../middleware/auth`"          |
| `preference`             | A coding style preference inferred from the team's revision feedback     | "This team rejects Promise chains — always use async/await"                   |
| `constraint`             | A hard rule about what NOT to do in this codebase                        | "Never drop the `ai_configurations` table — it's owned by Story 2.5"          |
| `architectural_decision` | A design choice made for the workspace that all future work must respect | "All DB queries go through Drizzle ORM repositories, never raw SQL"           |
| `checkpoint`             | Partial task state for a Jira story that was interrupted                 | "PROJ-42 half-done: A.ts and B.ts modified, tests passing, C.ts still needed" |

## Acceptance Criteria

### Database

- [ ] `workspace_memory` table:

  | Column             | Type                                                                                             |
  | ------------------ | ------------------------------------------------------------------------------------------------ |
  | id                 | UUID PK DEFAULT gen_random_uuid()                                                                |
  | workspace_id       | UUID FK → workspaces.id ON DELETE CASCADE                                                        |
  | type               | VARCHAR(30) CHECK IN ('pattern','preference','constraint','architectural_decision','checkpoint') |
  | key                | VARCHAR(300) NOT NULL — short identifier for this memory (e.g. "api-route-import-pattern")       |
  | content            | TEXT NOT NULL — the full memory text the AI reads                                                |
  | confidence         | NUMERIC(3,2) DEFAULT 1.0 CHECK (confidence BETWEEN 0 AND 1)                                      |
  | source_workflow_id | UUID FK → workflow_executions.id NULL ON DELETE SET NULL                                         |
  | jira_story_id      | VARCHAR(100) NULL — if memory is story-scoped (checkpoints), else NULL for workspace-wide        |
  | created_at         | TIMESTAMPTZ DEFAULT NOW()                                                                        |
  | updated_at         | TIMESTAMPTZ DEFAULT NOW()                                                                        |

- [ ] Unique constraint on `(workspace_id, type, key)` — prevents duplicate memories for the same concept. Upserting on conflict updates `content`, `confidence`, and `updated_at`.
- [ ] Index on `(workspace_id, type)` for fast retrieval by type at bootstrap time.
- [ ] Index on `(workspace_id, jira_story_id)` for fast checkpoint lookup.

### Repository

- [ ] `MemoryRepository`:
  - `upsert(data)` — insert or update on `(workspace_id, type, key)` conflict.
  - `findByWorkspaceAndType(workspaceId, type)` — returns all memories of a given type for the workspace.
  - `findCheckpointByStory(workspaceId, jiraStoryId)` — returns the latest `checkpoint` memory for the story, or null.
  - `delete(id)` — hard delete by id (for admin UI cleanup).
  - `deleteByWorkspaceAndType(workspaceId, type)` — bulk clear (e.g., reset all preferences).

### Admin API

- [ ] `GET /api/workspaces/:id/memory` — list all memories (paginated, filterable by `?type=preference`). WORKSPACE_OWNER+ only.
- [ ] `DELETE /api/workspaces/:id/memory/:memoryId` — delete a single memory. WORKSPACE_OWNER+ only.
- [ ] `DELETE /api/workspaces/:id/memory?type=preference` — bulk delete by type. WORKSPACE_OWNER+ only. Requires confirmation token in body: `{ confirm: "DELETE_ALL_preference" }` to prevent accidental data loss.
- [ ] `PUT /api/workspaces/:id/memory/:memoryId/confidence` — update confidence score (workspace owner can promote or demote an auto-learned memory). WORKSPACE_OWNER+ only.

## Tests to Write

- `upsert()` new memory → stored in DB.
- `upsert()` same `(workspace_id, type, key)` → updates content, not duplicate row.
- `findByWorkspaceAndType()` → returns only memories for that workspace and type.
- `findCheckpointByStory()` → returns latest checkpoint for story, null if none.
- `delete()` → row removed.
- `GET /memory` with DEVELOPER role → 403.
- `GET /memory?type=preference` → returns only preference memories.
- Bulk delete without confirmation token → 400.
- Bulk delete with correct confirmation token → all matching rows deleted.
