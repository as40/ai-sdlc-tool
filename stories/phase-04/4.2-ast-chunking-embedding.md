`[ ] PENDING`

## Description
Build a background worker that iterates through cloned files, parses them with AST chunkers to split by logical blocks (functions/classes), calls the configured embedding model, and batch-inserts vectors into PostgreSQL.

## Acceptance Criteria
- [ ] `EmbeddingWorker` runs in a Node.js `worker_thread` (not blocking the main event loop).
- [ ] File traversal:
  - Respects `.gitignore` rules (use `ignore` npm package).
  - Skips binary files, `node_modules/`, `dist/`, `*.lock` files.
  - Processes: `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.java`, `.go`, `.rs`, `.rb`, `.cs`, `.md`.
- [ ] Chunking strategy:
  - **TypeScript/JavaScript**: Use `@typescript-eslint/parser` to identify function and class boundaries. Each function/class = one chunk.
  - **Other languages**: Fallback to sliding window chunking (512 tokens, 50 token overlap).
  - Max chunk size: 1500 tokens (truncate if needed).
- [ ] File hash check: if `SHA-256(fileContent) === stored file_hash` for a chunk, skip the file entirely (no re-embedding).
- [ ] Batch embedding: accumulate up to 50 chunks, then send a single batch embedding request to the configured model.
- [ ] Batch DB insert: use `EmbeddingRepository.upsertChunk()` for each chunk in the batch.
- [ ] Progress events emitted via WebSocket: `{ type: 'vectorization_progress', payload: { repoId, filesProcessed, totalFiles, chunksCreated } }`.
- [ ] Worker triggered via `POST /api/workspaces/:id/repos/:repoId/vectorize`.
- [ ] `GET /api/workspaces/:id/repos/:repoId/vectorize/status` — returns current progress.

## Technical Notes
- Embedding model called via `LLMGateway` service (stub from 2.5).
- If embedding API rate limit hit: exponential backoff, max 3 retries.
- `worker_threads` communicate with main thread via `MessageChannel`.
- Token counting: use `tiktoken` or `@anthropic-ai/tokenizer` for rough token estimates.
