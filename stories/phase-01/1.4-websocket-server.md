# Story 1.4 — WebSocket Server Implementation

## Phase
Phase 1: Foundation & Platform Scaffolding

## Dependencies
- Story 1.1 must be `DONE`.

## Status
`[ ] PENDING`

## Description
Integrate a typed WebSocket server into the Express application using the `ws` library. This server will be used across later phases for streaming code generation, vectorization progress, container logs, and workflow state updates.

## Acceptance Criteria
- [ ] WebSocket server is attached to the same HTTP server instance as Express (shared port).
- [ ] Connection upgrade handled cleanly — unauthenticated WS connections are rejected with `401` during the HTTP upgrade handshake (JWT validated from query param or `Authorization` header).
- [ ] A `WebSocketService` class manages active connections keyed by `userId`.
- [ ] Public method: `broadcast(userId: string, event: WsEvent): void` — sends a typed event to all connections for that user.
- [ ] Public method: `broadcastToWorkspace(workspaceId: string, event: WsEvent): void` — sends to all users in a workspace.
- [ ] All messages are typed JSON: `{ type: string; payload: unknown; timestamp: string }`.
- [ ] Incoming messages from clients are validated with Zod before processing.
- [ ] Ping/pong heartbeat every 30 seconds; dead connections are terminated and removed from the connection map.
- [ ] Graceful shutdown: on `SIGTERM`, close all WS connections before shutting down the HTTP server.
- [ ] A test client can connect, send `{ type: 'ping' }`, and receive `{ type: 'pong' }` back.

## Technical Notes
- Use the `ws` npm package (not `socket.io` — lighter weight, matches the PRD spec).
- Attach ws server: `new WebSocketServer({ server: httpServer })`.
- JWT auth on upgrade: extract token from `req.headers['authorization']` or `?token=` query param; verify with `jsonwebtoken.verify`.
- Connection map: `Map<string, Set<WebSocket>>` where key is `userId`.
- Event types defined as a discriminated union in `src/types/ws-events.ts`.
