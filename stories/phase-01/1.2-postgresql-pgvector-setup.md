# Story 1.2 — Setup PostgreSQL & pgvector Database

## Phase
Phase 1: Foundation & Platform Scaffolding

## Dependencies
- Story 1.1 must be `DONE`.

## Status
`[ ] PENDING`

## Description
Spin up a local PostgreSQL instance with the pgvector extension and write the initial database migrations for the core entities that every subsequent phase will build on.

## Acceptance Criteria
- [ ] `docker-compose.yml` (dev only) spins up PostgreSQL 16 with `pgvector` extension available.
- [ ] `pgvector` extension enabled in the database via migration.
- [ ] Drizzle ORM configured in `packages/backend/src/db/` with connection pooling (`pg` pool).
- [ ] Migration runner command: `pnpm --filter backend db:migrate`.
- [ ] Schema seed command: `pnpm --filter backend db:seed` (optional, for dev seed data).
- [ ] The following tables exist after migrations run:

### `users`
| Column | Type | Constraints |
|--------|------|-------------|
| id | UUID (v7) | PK |
| email | VARCHAR(255) | UNIQUE NOT NULL |
| display_name | VARCHAR(255) | NOT NULL |
| access_level | ENUM | NOT NULL DEFAULT 'DEVELOPER' |
| created_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() |
| updated_at | TIMESTAMPTZ | NOT NULL DEFAULT NOW() |
| deleted_at | TIMESTAMPTZ | NULL |

### `workspaces`

## Status Enums
- `access_level`: `SUPER_ADMIN`, `WORKSPACE_OWNER`, `DEVELOPER`, `VIEWER`
- `workflow_status`: `IDLE`, `RUNNING`, `AWAITING_APPROVAL`, `REVISION_REQUESTED`, `COMPLETED`

## Technical Notes
- Use Drizzle ORM schema definitions (TypeScript) as the source of truth.
- Migration files generated by `drizzle-kit generate`.
- Connection string via `DATABASE_URL` env var.
- All FK constraints with `ON DELETE RESTRICT` unless otherwise specified.
- `updated_at` auto-updated via a DB trigger or Drizzle `$onUpdate`.
