## Description

When the AI calls `request_human_input`, render an interactive action block directly inside the chat thread rather than a disruptive modal. The block shows the agent's question, optional context, and clickable option buttons. Selecting an option dispatches the answer to the backend, resumes the workflow, and locks the block to show what was chosen. This mirrors the pattern used by Claude Code and GitHub Copilot where the AI's question is a first-class conversation turn.

## Acceptance Criteria

- [ ] When a `workflow_state_change` event arrives with `status: 'AWAITING_INPUT'` AND a matching `agent_activity` event with `activity_type: 'hitl_pending'`, the chat panel inserts a `HitlActionBlock` component as the last message in the thread.
- [ ] The frontend fetches the HITL request payload from `GET /api/workspaces/:wsId/workflows/:workflowId/pending-input` to get the full `{ message, options, context, allow_free_text }` payload.
  - Backend endpoint added in this story. Auth: JWT + workspace membership.
  - Returns 404 if no pending HITL request exists.
  - Response shape: `{ pendingId: string; toolName: string; proposedOutput: unknown; sessionId: string }` — derived from the stored `payload` in the registry.
- [ ] `HitlActionBlock` renders:
  - Agent name badge.
  - `message` text.
  - `context` text in a muted code-block style (if present).
  - One button per entry in `options` (max 4 rendered; overflow hidden).
  - A free-text input + "Send" button if `allow_free_text: true`.
- [ ] Clicking an option button:
  1. Disables all buttons immediately (prevents double-submission).
  2. `POST /api/workspaces/:wsId/workflows/:workflowId/respond` with `{ answer: selectedOption }`.
  3. On success: replace buttons with `"✔ You chose: {option}"` and a timestamp.
  4. The `hitl_pending` activity chip in the activity strip is cleared (via `workflow_state_change: RUNNING`).
- [ ] Submitting a free-text reply:
  1. Same flow as clicking an option, with `answer` set to the input text.
  2. Validation: non-empty, max 2000 chars (matches backend Zod schema).
- [ ] If the HITL request times out before the user responds (backend emits `workflow_state_change: IDLE` with error `HITL_TIMEOUT`):
  - All buttons and input are disabled.
  - Block shows: `"⚠ Request timed out — workflow stopped."`.
- [ ] The action block is NOT a modal — it does not block the rest of the UI. The user can still read earlier chat messages, inspect the code editor, and view the file tree while a HITL request is pending.
- [ ] A workflow can only have one active HITL request at a time. The block is unique per `workflowId`. Subsequent `AWAITING_INPUT` events for the same workflow replace the previous block.

## Technical Notes

- `GET /api/workspaces/:wsId/workflows/:workflowId/pending-input` reads from `PendingInputRegistry` (Story 5.7). For this endpoint to work, the registry must store the original HITL request payload (not just resolve/reject callbacks) — see Story 5.7 Technical Notes for the updated registry shape: `Map<string, { resolve, reject, payload: HITLRequestPayload }>`.

## Tests to Write

- `workflow_state_change: AWAITING_INPUT` → `HitlActionBlock` appended to chat messages.
- Block renders `message`, `context`, and correct number of option buttons.
- Clicking button calls respond endpoint with correct `answer`.
- Double-click after first click → second call NOT made (buttons disabled).
- On success response → buttons replaced with confirmation text.
- Free-text submit with empty string → validation error, no API call.
- `workflow_state_change: IDLE` with `HITL_TIMEOUT` → block shows timeout message.
- 404 from `pending-input` endpoint → block shows graceful error, no crash.

## Definition of Done

- HITL questions render as interactive chat blocks with option buttons. Selection dispatches answer and locks the block. Timeout handled gracefully. No modal — fully inline in the chat thread.
