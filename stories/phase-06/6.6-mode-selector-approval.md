# Story 6.6 — Mode Selector & Approval Control Panel

## Phase

Phase 6: Visual Workspace IDE & Live Code Editor

## Dependencies

- Story 5.5, 6.1 must be `DONE`.

## Status

`[ ] PENDING`

## Description

Add a global Auto/Manual mode toggle in the IDE header and design the AWAITING_APPROVAL modal overlay with Approve and "Reject & Revise" actions.

## Acceptance Criteria

- [ ] Toggle switch in TopNavBar to switch between `AUTO` and `MANUAL` execution modes.
- [ ] Mode stored in workspace settings (`PUT /api/workspaces/:id/settings { executionMode }`).
- [ ] When a workflow enters `AWAITING_APPROVAL` state (received via WebSocket):
  - A modal overlay appears blocking the IDE.
  - Modal shows: agent name, summary of what was done, the git diff (from 6.5).
  - Two buttons: "Approve" (calls `POST .../approve`) and "Reject & Revise" (reveals text area).
  - Reject path: user types feedback in textarea, submits (calls `POST .../revise`).
  - Modal dismisses and workflow resumes.
- [ ] In `AUTO` mode, the modal never appears (orchestrator auto-approves).
- [ ] "Reject & Revise" textarea: minimum 20 characters enforced (Zod on frontend).
- [ ] Revision count shown in modal: "Revision 2 of 5".
- [ ] The 20-character minimum on revision feedback is enforced on BOTH frontend (Zod schema on submit) AND backend (`POST .../revise` validates `feedback` length ≥ 20 with Zod, returns 422 with `{ status: 422, title: 'Validation Error', detail: 'Feedback must be at least 20 characters' }`).
- [ ] Approve and Revise buttons show loading state ('Approving…'/'Submitting…') while API call is in flight. Button disabled during loading.

- WS event `awaiting_approval` opens modal.
- Approve button calls approve endpoint and closes modal.
- Revise with < 20 chars shows validation error.
- Revise with valid feedback calls revise endpoint.
