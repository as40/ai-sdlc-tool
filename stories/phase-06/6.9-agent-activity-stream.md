## Description

Add a live activity strip above the chat window that shows exactly what the agent is doing at any given moment. When a tool call starts the strip shows a chip ("Reading src/index.ts..."); when it finishes the chip fades out. This gives the operator Claude Code / Copilot-style visibility into background agent operations without cluttering the main conversation history.

## UI Behaviour

```
┌──────────────────────────────────────────────────────────────┐
│  [⚙ Running tests...]  [📄 Reading src/auth/jwt.ts...]       │  ← activity strip
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  [Chat message history...]                                   │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

- Chips are transient: each chip auto-removes 3 seconds after its matching `tool_done` / `tool_error` event arrives.
- At most 4 chips are shown simultaneously. Overflow is hidden (not queued).
- When no agent is running, the strip is hidden (zero height, no layout shift).
- Chips are colour-coded by `activity_type`:
  - `llm_thinking` → gray pulse animation
  - `tool_start` → blue
  - `tool_done` → green (brief flash before removing)
  - `tool_error` → red
  - `hitl_pending` → amber (persists; removed when HITL is resolved)

## Acceptance Criteria

- [ ] `AgentActivityStrip` component listens to the existing WebSocket connection (from `useWorkflowWebSocket` hook).
- [ ] On `agent_activity` event: adds a chip to the strip with `description` as label and appropriate colour.
- [ ] On `tool_done` / `tool_error` events: the matching `tool_start` chip transitions to done/error colour then removes after 3 s.
- [ ] `hitl_pending` chip stays visible until the workflow transitions out of `AWAITING_INPUT` (next `workflow_state_change` event resolves it).
- [ ] Chips are animated in/out using CSS transitions (no external animation library).
- [ ] Each chip shows a spinner icon while active and a checkmark/X on completion.
- [ ] Strip is fully keyboard-accessible (chips are `role="status"` live regions for screen readers).
- [ ] Activity events do NOT appear as messages in the chat history — the two streams are separate.
- [ ] When the workflow is `COMPLETED` or transitions to `IDLE`, all remaining chips are cleared.
- [ ] `useWorkflowWebSocket` hook lives at `packages/frontend/src/hooks/useWorkflowWebSocket.ts`. It opens a WebSocket connection to `ws://localhost:{PORT}/ws` with `?token={jwt}` query param for auth, and provides `subscribe(eventType, handler)` / `unsubscribe()` methods. This is the shared WS hook used by all IDE panels.

## Tests to Write

- `agent_activity` event with `tool_start` → chip appears with correct label and colour.
- Matching `tool_done` arrives → chip transitions to green, removed after 3 s.
- `tool_error` arrives → chip turns red.
- `hitl_pending` chip is NOT auto-removed after 3 s.
- `hitl_pending` chip removed when `workflow_state_change` with status `RUNNING` arrives.
- More than 4 simultaneous chips → 5th not rendered.
- Workflow transitions to `COMPLETED` → all chips cleared.
- No chip rendered when no workflow is active.

## Definition of Done

- Activity strip renders live `agent_activity` events as transient chips. Chips have correct lifecycle: appear, change state, auto-remove. Strip hidden when idle. Does not pollute the chat message list.
