# Story 12.11 — Preference Learning from Revision Feedback

## Phase

Phase 12: Session Isolation & Memory Lifecycle

## Dependencies

- Story 4.4 must be `DONE` (`workspace_memory` table required).
- Story 5.5 must be `DONE` (revision feedback is submitted via the HITL revisor).
- Story 12.10 must be `DONE` (establishes the memory write pattern this story extends).

## Status

`[ ] PENDING`

## Description

When a human rejects the AI's output and provides revision feedback (Story 5.5), the feedback often contains implicit team preferences: "never use any type", "always add a migration when changing the schema", "don't mock the database in tests". These preferences are currently lost at session end and the AI repeats the same mistakes in the next session.

This story adds a lightweight extraction pass that runs on every piece of revision feedback, detects preference signals, and saves them to `workspace_memory` as `preference` entries. These memories are then included in every future session bootstrap, permanently shaping the AI's behaviour for this workspace without requiring the human to repeat the same correction.

## Acceptance Criteria

### Preference Extraction

- [ ] `PreferenceLearner` service with method `extract(feedback: string, workspaceId: string, workflowExecutionId: string): Promise<void>`.
- [ ] Called by the HITL revisor (Story 5.5) immediately after a revision is submitted, as a non-blocking background task. If extraction fails, log a warning — it must NOT block or fail the revision flow.
- [ ] Extraction uses a lightweight LLM call (economy tier model via `LLMGateway`):
  - System prompt: `"You are a preference extractor. Given a code review comment, identify any explicit coding preferences, constraints, or style rules. Return a JSON array of { key: string, preference: string } objects. If none exist, return []. Keys should be short kebab-case slugs. Return ONLY valid JSON."`
  - Input: the revision feedback text.
  - Output: parsed JSON array or empty array on failure.
- [ ] For each extracted preference:
  - Call `MemoryRepository.upsert()` with `type: 'preference'`, `key: {extracted_key}`, `content: {preference}`, `confidence: 0.7`, `source_workflow_id`.
  - Confidence is set to `0.7` (auto-learned, not manually confirmed). Workspace owner can raise it to `1.0` via the admin UI (Story 4.4).
- [ ] Extraction is skipped (no LLM call) if feedback is under 10 characters — too short to contain a preference.
- [ ] Extraction is skipped if the same `(workspace_id, type, key)` already exists at `confidence >= 1.0` — human-confirmed memories are not overwritten by auto-learning.

### Confidence Escalation Signal

- [ ] A preference that has been auto-learned from 3 or more distinct revision events (determined by counting `source_workflow_id` diversity — queried from `workspace_memory`) is automatically promoted from `confidence: 0.7` to `confidence: 0.9`.
- [ ] Workspace owners see a visual indicator in the memory admin UI (Story 4.4 API `GET /memory`) distinguishing:
  - `confidence < 0.8`: auto-learned, unconfirmed — shown with a note "auto-detected, not confirmed"
  - `confidence 0.8–0.99`: reinforced by multiple sessions — shown with "seen N times"
  - `confidence = 1.0`: manually confirmed by workspace owner — shown as "confirmed"

### Integration with Context Bootstrap

- [ ] Story 12.4 (context bootstrapper) is updated in this story to include a **Workspace Preferences** section in the bootstrap message when `preference` memories exist:
  ```
  WORKSPACE PREFERENCES (learned from team feedback):
  - {preference_1}
  - {preference_2}
  ...
  These are not rules from the rules files — they are patterns this team has corrected the AI on before.
  Follow them unless the story explicitly requires otherwise.
  ```
- [ ] Preferences with `confidence < 0.5` are excluded from the bootstrap (too uncertain to be useful).
- [ ] Preferences section is omitted entirely if no preferences exist (no empty block).

## Technical Notes

- The LLM call for extraction uses the workspace's `economy` tier model — this is a simple classification task, not a reasoning task.
- The extraction is intentionally lightweight: one turn, no tool calls, structured JSON output. If the model returns malformed JSON, catch the parse error, log it, and return without saving.
- Extraction runs asynchronously — use `setImmediate()` or a detached Promise after the revision response is sent to the client. Never `await` it in the request handler.
- Keep the extraction system prompt short and the output format strict. Verbose extraction prompts cost tokens on every revision; this must be cheap.

## Tests to Write

- Feedback with "never use any type" → `preference` memory written with key `no-any-type`.
- Feedback with no preference signals ("looks good") → no memory written, no LLM call.
- Feedback under 10 characters → extraction skipped.
- Extraction LLM returns malformed JSON → warning logged, no crash, no memory written.
- Same key already at `confidence: 1.0` → upsert skipped.
- Same preference extracted from 3 distinct workflows → confidence auto-promoted to 0.9.
- Bootstrap with preferences present → `WORKSPACE PREFERENCES` section included.
- Bootstrap with no preferences → section omitted.
- Preferences with `confidence < 0.5` → excluded from bootstrap.
- Extraction failure does NOT fail or delay the revision API response.

## Definition of Done

- Revision feedback triggers preference extraction (non-blocking). Extracted preferences saved to workspace memory with correct confidence. Reinforced preferences auto-promoted. Preferences surface in all future session bootstraps. Tests pass.
