## Description

When a workflow is interrupted mid-execution (process restart, user closes the browser, network loss, or explicit cancel), the AI currently starts from scratch on the next run for the same Jira story. This wastes tokens re-discovering what was already done and can cause duplicate file writes or conflicting changes.

This story writes a **checkpoint** to `workspace_memory` after every completed workflow step, capturing the exact execution state. When a new workflow starts for the same Jira story, the orchestrator detects the checkpoint and offers the user a resume option via HITL before beginning fresh.

## Acceptance Criteria

### Checkpoint Writing

- [ ] After each workflow step completes (a full agent turn ending with `tool_done` or `AWAITING_APPROVAL`), the orchestrator writes a checkpoint via `MemoryRepository.upsert()`:
  - `type: 'checkpoint'`
  - `key: 'workflow-checkpoint-{jiraStoryId}'`
  - `jira_story_id: jiraStoryId`
  - `content`: JSON-serialised object:
    ```typescript
    {
      interrupted_at: string;         // ISO timestamp
      workflow_execution_id: string;
      completed_steps: string[];      // short description of each completed step
      files_modified: string[];       // all file paths written by write_file skill
      tests_status: 'passing' | 'failing' | 'not_run';
      last_agent_output: string;      // truncated to 500 chars
      remaining_hint: string;         // AI's own summary of what's still left (from its last message)
    }
    ```
  - Upserts on `(workspace_id, type, key)` — always overwrites the previous checkpoint for this story.
- [ ] Checkpoint is also written when a workflow transitions to `IDLE` via cancel or error (so interrupted workflows have a checkpoint too).
- [ ] Checkpoint is **deleted** (via `MemoryRepository.delete()`) when a workflow reaches `COMPLETED` — no stale checkpoints for finished stories.

### Resume Detection at Workflow Start

- [ ] In `POST /api/workspaces/:id/workflows`, before starting a new workflow:
  1. If `jiraStoryId` is provided: call `MemoryRepository.findCheckpointByStory(workspaceId, jiraStoryId)`.
  2. If a checkpoint exists: do NOT immediately start the workflow. Instead, transition to `AWAITING_INPUT` and emit a HITL event:
     ```
     A previous session for {JIRA-KEY} was interrupted.
     Progress so far:
     - Steps completed: {completed_steps}
     - Files modified: {files_modified}
     - Tests: {tests_status}
     - What's left: {remaining_hint}

     Do you want to resume from this checkpoint or start fresh?
     Options: ["Resume from checkpoint", "Start fresh (discard checkpoint)"]
     ```
  3. On "Resume from checkpoint": inject a resume context message as the first user message in the new session, summarising the checkpoint data. The checkpoint is preserved in `workspace_memory` until the new workflow completes.
  4. On "Start fresh": delete the checkpoint and start the workflow normally.
- [ ] If no checkpoint exists for the story: start normally without any HITL pause.
- [ ] Checkpoint detection only fires when `jiraStoryId` is present in the request. Workflows without a Jira story never check for checkpoints.

### Resume Context Injection

- [ ] When resuming, the following is prepended to the session's initial messages (before the bootstrap message from 12.4):
  ```
  RESUME CONTEXT for {JIRA-KEY}:
  You previously worked on this story and were interrupted. Here is what was accomplished:
  - Completed steps: {completed_steps}
  - Files already modified: {files_modified} — DO NOT rewrite these unless there is a specific reason.
  - Test status at interruption: {tests_status}
  - What remained: {remaining_hint}

  Continue from where you left off. Do not repeat work already done.
  ```
- [ ] The resume context message counts toward session `token_count`.

## Technical Notes

- The checkpoint `content` field is stored as a JSON string in the `TEXT` column. Parse it with `JSON.parse()` when reading. Use `JSON.stringify()` when writing.
- `files_modified` is accumulated by the orchestrator by collecting all `write_file` skill calls across all steps in the workflow. This list must be maintained in-memory on the orchestrator instance and persisted to the checkpoint on each step boundary.
- `remaining_hint` is extracted from the AI's last assistant message in the current session. Truncate to 500 characters to keep the checkpoint compact.
- The checkpoint HITL uses the existing `AWAITING_INPUT` state from Story 5.8 — no new states needed.

## Tests to Write

- Workflow step completes → checkpoint written to `workspace_memory`.
- Workflow cancelled → checkpoint written.
- Workflow completes → checkpoint deleted.
- New workflow starts for same story WITH checkpoint → HITL offered with resume details.
- New workflow starts for same story WITHOUT checkpoint → starts normally.
- New workflow without jiraStoryId → no checkpoint check performed.
- User chooses "Resume" → resume context prepended to session, checkpoint preserved.
- User chooses "Start fresh" → checkpoint deleted, workflow starts normally.
- Resume context token count included in session total.

## Definition of Done

- Interrupted workflows leave a checkpoint. New workflows for same story detect and offer resume. Completed workflows clean up their checkpoint. Tests pass.
